# 2026-07-11

## Two corrections after review: DirectEffort goes remote, NaN stops at the door

Yesterday's D-025 batch committed, then two of its conservative choices
were overruled on review.

DirectEffort over Keelson. The adapter had deliberately refused to
serialize Setpoint::DirectEffort on the theory that raw force demands
are local to the conn node. Overruled: direct remote control is a
supported story, a teleoperator streams DirectEffort like any other
setpoint under the same conn/arming authority. The setpoint proto gains
a direct_effort variant (surge N, sway N, yaw Nm), both conversion
directions are wired, and setpoint_to_proto is infallible again. D-025
itself never claimed wire-locality, so no decisions amendment; the
locality note in yesterday's diary entry is superseded by this one.

Non-finite values are now rejected at the boundaries instead of being
caught after the damage. Three seams, one policy:

- Wire: every numeric field of every incoming setpoint variant is
  finiteness-checked at proto-to-contract conversion
  (Error::NonFinite with a field path). A remote client streaming NaN
  tau never reaches guidance. Observed while there: no range validation
  exists beyond finiteness (lat/lon/heading unbounded); left as is,
  worth a look when the GNSS driver lands.
- Estimator intake: Rejection::NonFinite ahead of all other checks,
  covering value and declared-std fields of every MeasurementKind. One
  bad sample costs nothing; state, staleness, and health untouched.
  The health Fault-on-NaN gate from yesterday stays as the backstop
  for arithmetic-originated non-finiteness and is now tested through a
  cfg(test) seam that poisons the state directly.
- Supervisor voltage: IEEE comparisons with NaN are false, so an
  unguarded NaN voltage silently disabled both voltage failsafes
  rather than tripping anything. A non-finite reading is now skipped
  in favor of the last good value, and never overwrites it.

The supervisor finding is the one to remember: NaN does not trip
thresholds, it makes them vanish. Every future comparison against
sensor-derived data needs either an intake guard or a finite-by-
construction argument.

## First water rehearsed on the desk (subagent run)

No hardware access for a while, so the remaining software-only Phase 6
items ran as a second subagent queue with one framing: rehearse first
water entirely on the desk, leaving physics as the only new variable
when hardware arrives.

- GNSS driver over 0183 in coxswain-drivers: byte-fed push API, GGA
  gated on fix quality {GPS, DGPS, RTK fixed, RTK float} with
  std = HDOP x UERE (default 5 m UERE, fallback std when HDOP absent),
  HDT with configured std. RMC/VTG parse but emit nothing: no SOG/COG
  MeasurementKind exists and RMC position would double-count GGA.
  Sentences stamp with the terminating byte's acquisition time; the
  bias is bounded by one sentence duration and documented.
- RC adapter: CRSF frames to typed events. Kill dominates takeover and
  suppresses effort without clearing takeover state; switch thresholds
  carry hysteresis; sticks map linearly outside a center deadband,
  clamped at configured maxima; sway is always zero (no stick).
- Serial actuator backend: $CXACT,<surge>,<sway>,<yaw>*HH once per
  tick, line rate as the dead-man keepalive. Hand-rolled no-alloc
  fixed-point rendering; non-finite demands refused at the last
  boundary before the wire. Golden lines re-verified through the 0183
  parser, so the framing provably matches a standard tokenizer.
- Hosted real-serial backend: --port maps the manifest's logical bus
  names to /dev paths (unmapped inner_loop bus is a boot error per
  D-009), --rc-port/--actuator-port for the conn-node-local links,
  termios raw mode via the already-transitive libc. contrib/
  coxswain.service documents deployment.
- Desk-rig integration tests over ptys, the exit criterion: real
  serial reads feed the estimator from a sim plant rendered as 5 Hz
  GGA+HDT; and the full authority story runs over pty CRSF: teleop
  arms, RC preempts by manifest priority, stick surge reaches the
  plant through $CXACT (1.23 m/s), kill zeroes demand within 200 ms,
  release keeps the conn without ClaimantLost. Five runs, no flakes.

Two honest gaps surfaced and recorded in docs/hardware.md and TASKS:
the termios path cannot set CRSF's 420000 baud (ptys mask it; needs
termios2/BOTHER), and no power monitoring input path exists in
real-serial mode, which blocks armed on-water operation. RC re-arm is
deliberately absent: kill is a one-way latch, re-arming takes another
claimant, which reads as a correct safety property rather than a gap.

docs/hardware.md drafted as the fence: interfaces supported by
construction, first-water requirements from yesterday's experiment,
an explicitly unverified procurement shortlist, and the gaps. Parts
can be ordered while hardware access is away.

## Third queue: both gaps closed, Phase 7 software items done

- termios2/BOTHER for rates outside the Bxxxx table (CRSF's 420000).
  A set-speed failure is a typed error except on a pty slave, where
  speed is meaningless; the pty check is a path rule because TIOCGPTN
  answers only on the master side (verified empirically, and on this
  kernel a pty actually accepts BOTHER, so the tolerance branch is a
  defensive fallback).
- Power input path settled and built: the actuator far end measures
  the battery and reports $CXPWR,<voltage_v>*HH at about 1 Hz on the
  actuator link's reverse direction, command-then-report lite under
  D-021 until Cyphal. Dedicated strict accumulator (the 0183
  SentenceReader has no hook for unknown addresses); unknown traffic
  on our own wire is ignored, unlike the strict GNSS path. Desk rig:
  arm refused two ticks after a low-voltage report with armed state
  surviving (report-only per matrix v1), critical voltage disarms.
  Report staleness is a recorded open item for the failsafe matrix,
  not improvised here.
- 0183-over-UDP: manifest-declared buses bind at boot, source_ip
  pinning enforced at the socket with drop counting, bytes ride the
  same gnss0183 path with one receive timestamp per datagram. Bound
  to 0.0.0.0 rather than SO_BINDTODEVICE (needs CAP_NET_RAW; D-014
  frames the pin as a topology assertion, not authentication). Test
  proves a spoofed 127.0.0.3 sender cannot move the estimate. One
  behavioral fix rode along: a UDP-only manifest now counts as real
  I/O instead of silently booting the simulator.
- coxswain-n2k: zero-dep no_std decode of the initial single-frame
  PGN set (heading, ROT, depth, position, COG/SOG, wind), sentinels
  to None, SI out, field definitions verified against canboat's
  pgn.h, two goldens from real canboat sample frames. Enrichment
  only per D-011; deliberately no MeasurementKind mapping.
  Fast-packet (129029) and hosted CAN wiring wait for hardware.

Software-side Phase 6 and 7 are now done except items that need a
bench or a device: IMU/mag drivers (off the critical path), SBF,
Cyphal, hosted CAN wiring, and the water itself. Next session with
hardware starts from a rehearsed rig, not from scratch.

## Allocation brought into scope: conn node, effector table (D-026)

The $CXACT link's "far end maps tau onto its actuators" clause was
carrying an unowned design problem, and the first real hull is
ESC + rudder, which that clause quietly made someone else's firmware.
Discussed and settled three things: allocation runs at the conn node
(allocation coefficients shape control, so D-018 puts them in the
signed blob and the audited core consumes them); v1 effector kinds are
fixed thruster and rudder, so one pseudo-inverse solver covers both
the ESC + rudder hull and twin differential, with layouts falling out
of the effector table rather than an enum; and the underactuated hold
mode lands in the same batch, because a rudder boat cannot hold a
point and ClaimantLost station-keep has to stay safe on it. The wire
format moves from tau ($CXACT) to per-channel outputs ($CXOUT) when
the allocator lands, making the far end a dumb watchdogged PWM bridge.

Sailing was looked at and deliberately kept out of the effector
matrix: sail force is wind-constrained, speed stops being a free
variable, and tacking sits above the control law, so a sailing vessel
class changes guidance (polar, no-go zone) plus a slow trim loop, not
allocation. Only obligation taken on now: the effector table must
tolerate effectors that do not consume tau.

Written as D-026, queued as Phase 6b in TASKS.md, effector geometry
recorded as manifest v0.4 open question 4. hardware.md keeps
documenting $CXACT until the code actually changes.

## Output backends: termination declared per effector (D-027)

Follow-on question: several actuator-end interfaces, direct PWM
servos, CAN actuators, and so on. Settled as symmetry with the sensor
side: effectors reference a manifest output bus the way sensors
reference input buses, one backend per bus kind. Three named: the
$CXOUT serial bridge (6b), Cyphal nodes (7, where the backend trait
crystallizes; one implementation is a single-use abstraction), and a
new "pwm" bus kind for conn-node timer pins.

The rule that fell out and earns the entry: every effector needs a
failsafe path that survives death of the control process. Bridge MCU
watches line silence, Cyphal node watches heartbeats, direct PWM has
only what the platform gives. On the H7 that is the hardware watchdog
plus safe timer defaults, so pwm lands with Phase 8; on Linux a dead
process leaves the peripheral holding its last duty cycle, so the
hosted profile refuses a pwm bus at boot. Directly connected on Linux
means the serial bridge, whose MCU is the watchdog.

Calibration placement per D-018: serial bridge is commanded in final
per-channel microseconds rendered at the conn node from manifest
calibration, so the bridge stays dumb and props change without a
reflash; Cyphal nodes take physical units and own their local
calibration, audited via command-then-report; direct PWM calibration
is manifest data by necessity, there is no far end.

One asymmetry worth remembering for Phase 8: PWM signal loss stops a
typical ESC but leaves a servo limp or holding position, so a
watchdog reset that merely silences the pins is a thrust failsafe and
not a rudder failsafe. The post-reset boot path must actively drive
the manifest-declared failsafe pulses. Recorded on the Phase 8 task.
