# 2026-07-13

## GNSS covariance over NMEA 0183 (GST), the vendor-neutral path

Landed real per-axis position covariance and RTK fix status from a plain
0183 receiver, replacing the HDOP*UERE scalar the estimator gets today.
Chosen over a Septentrio SBF binary decoder because the estimator benefit
(covariance + fix mode, consumed since the SOG/COG commit) is reachable over
standard 0183: the GST sentence carries per-axis error stats and GGA quality
already flags RTK fixed/float. SBF stays deferred for its receiver-specific
extras (PPS, dual-antenna heading). Recorded in TASKS Phase 7.

### The pairing problem

GST carries the error statistics but no position; GGA carries position and
fix quality but only HDOP. So `GnssPositionCov` needs both sentences, and
their order within an epoch is receiver-specific (some emit GST before GGA,
some after). Design landed: the driver keeps the most recent usable GST
covariance and pairs it with the next GGA if that GST is within `GST_MAX_AGE`
(1.5 s); otherwise the fix falls back to the scalar `GnssPosition`. Position
is always the current GGA's, so only the covariance can lag, and by at most
one epoch on a 1 Hz receiver. The fix mode comes from the current GGA, so an
RTK fixed<->float transition is reflected immediately even if the paired
covariance lags a tick.

The fallback degrades cleanly across the three cases that matter:
- receiver never emits GST -> always scalar, zero added latency (today's
  behaviour, unchanged);
- GST occasionally dropped -> the previous GST carries until it ages out of
  the window, then scalar;
- GST every epoch -> `GnssPositionCov` every fix.

A GST that parses but reports no usable stats (all fields empty) does not
overwrite a good prior one; it just ages out. GST is opt-in per the manifest
sentence list, so a manifest that does not list GST keeps scalar behaviour.

### Covariance from GST: diagonal, not the ellipse

GST reports both an error ellipse (semi-major, semi-minor, orientation) and
per-axis lat/lon stds. The ellipse carries the N/E cross-covariance; the
per-axis stds are the covariance diagonal. Used the diagonal
(`[[std_lat^2, 0], [0, std_lon^2]]`) and dropped the cross term: recovering it
from the ellipse needs sin/cos, which are not in `no_std` without a libm
dependency, and the cross term is a second-order refinement over the correct
per-axis variances. The diagonal is already anisotropic, unlike the isotropic
HDOP scalar it replaces. If a rotated-ellipse case ever proves it matters,
libm + the rotation is a small, contained change.

The parser still surfaces the full GST (ellipse fields included); the driver
chooses the diagonal subset, same split as GGA (parser surfaces satellites
and altitude the driver does not consume).
