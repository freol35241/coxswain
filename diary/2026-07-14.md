## Cyphal actuator backend: increment 2c plan (handoff)

Increments 1, 2a, 2b are done and pushed on branch `cyphal-actuator-backend`
(coxswain-cyphal transport crate; the OutputBackend trait + $CXOUT refactor to
the physical-units boundary, D-028; the CyphalActuatorBackend with
command-then-report). 2c is the hosted integration, one coupled unit, to be
done as its own focused session. Four parts:

1. **Effector schema (signed blob, schema_version 4 -> 5).** An effector's
   output wiring depends on its bus kind. Cleanest shape: an `EffectorOutput`
   enum, `Serial { channel, pwm }` vs `Cyphal { node_id, command_subject,
   feedback_subject }`, selected by the effector's bus kind at compile.
   Validation: a `cyphal_can` effector requires the Cyphal fields (not
   channel/pwm); an `actuator_uart`/`pwm` effector requires channel/pwm (not
   subjects). Blast radius: bump schema_version, update every golden/tamper/
   version-rejection test, the manifest schema doc, and the blob reader; add a
   `cyphal_can` effector to a test manifest. Record as a DECISIONS entry
   (D-029, or a D-028 follow-on) BEFORE coding, since it changes the blob
   format (D-018).

2. **Read-write CAN transport (`hosted/src/can.rs`).** Today it is listen-only
   by construction (N2K, D-011), with an explicit "no write path" invariant.
   The Cyphal control bus is D-011's transmit-allowed exception: add a
   `write_frame` and a read-write open, and update the module doc to carve out
   that exception (keep the listen-only path for N2K).

3. **Hosted wiring.** For a `cyphal_can` bus carrying effectors, build a
   `CyphalActuatorBackend` from the manifest (per-effector CyphalEffector
   subjects, conn node id, power subject, tolerance), select it as that bus's
   output backend, drive it through the trait with a CanSink writing frames,
   and read feedback via `can::spawn_reader` -> `handle_frame` -> `core.power`

   divergence into supervisor health.

4. **vcan rig (`hosted/tests/cyphal_can_rig.rs`), mirroring can_rig.rs.** The
   test plays the actuator node: reads command frames off vcan, sends feedback
   frames back, and asserts the command-then-report result and power reach the
   observable surface (health telemetry). Napkin-scale byte tests run
   everywhere; the vcan end-to-end test skips gracefully where vcan is
   unavailable (the require_vcan pattern).

Kick-off for next session: check out `cyphal-actuator-backend`, then start with
part 1 (the schema DECISIONS entry, then the schema_version bump), verifying at

## Cyphal actuator backend: increment 2c done

2c landed. The three open schema decisions from the handoff went, after
discussion with the operator: (1) the conn node's own bus id lives as `node_id`
on the `cyphal_can` `[[bus]]`, and the power node's voltage subject as `subject`
on the role=power `[[sensor]]`; (2) `report_tolerance` is per effector, in that
effector's physical units (a single bus scalar cannot be right for both a
thruster in newtons and a rudder in radians); (3) effectors and
`[[actuator_node]]` stay mutually exclusive, as on the serial path. All three
are recorded in D-029, written before the code.

What shipped:

- Schema v4 -> v5. `EffectorEntry.output` is now an `EffectorOutput` sum type,
  `Serial { channel, pwm }` vs `Cyphal { node_id, command_subject,
  feedback_subject, report_tolerance }`, selected by the effector's bus kind at
  compile; the wrong arm's fields are rejected. `BusEntry.node_id` and
  `SensorEntry.subject` added. coxswain-manifest now depends on coxswain-cyphal
  for the node-id/subject-id wire ranges (single source of truth). New
  `cyphal_effector.rs` golden + one rejection per new rule.
- Backend refactor: `report_tolerance` moved onto `CyphalEffector`;
  `power_subject` is now `Option` (the failsafe matrix already tolerates no
  power link, so a Cyphal actuator bus need not have a power node).
- `hosted/src/can.rs` gained `write_frame`; the module's listen-only invariant
  is carved to the N2K path, with the Cyphal control bus as D-011's
  transmit-allowed exception.
- Hosted wiring: a `CyphalActuatorLink` per `cyphal_can` bus carrying
  effectors, built from the manifest, driven through the output backend trait
  with a `CanSink`, feedback/power read via `can::spawn_reader` ->
  `handle_frame`. Power feeds the existing `PowerStatus` intake; per-effector
  divergence and the supervisor's report-only low-voltage flag surface in the
  published health as `actuation` and supervisor `power` subjects. A new
  `ClaimantClient::subscribe_health` distills both, so a claimant observes them
  without arming.
- vcan rig `hosted/tests/cyphal_can_rig.rs`, mirroring can_rig.rs: byte-level
  napkin tests run everywhere; the end-to-end test plays the actuator/power
  node and asserts command frames, divergence, and voltage reach health, and
  skips gracefully where vcan is unavailable. A no-vcan boot-error test drives
  the real binary through the whole Cyphal boot path (an unopenable control bus
  fails boot).

Not done, deferred honestly: node-local failsafe/heartbeat for a Cyphal

## Schema UX review: two enforcement gaps closed, two open questions raised

Went over the manifest TOML as the single point of contact with the
commissioning engineer, judging it for coherence rather than correctness. The
spine holds up (identity -> board profile -> buses -> things that reference
buses -> policy; string-id references; `license` as the one trust bit; unit
suffixes; defaults for plumbing but never for trust or physics). Two settled
invariants turned out not to be enforced, and two genuine consistency questions
surfaced that want deciding before the next `schema_version` bump.

Enforcement gaps closed now (both enforce already-settled decisions, no schema
change, no version bump):

- **`[rc].claimant` must name a declared `[[claimant]]` (D-025).** The RC's
  preemption priority lives in `[[claimant]]`, keyed by that integer. A `[rc]`
  pointing at an id no claimant declares compiled cleanly and the RC silently
  ran at the default priority 0, the opposite of "the hand controller outranks
  autonomy". Now a `ValidateError::RcClaimantUnknown`.
- **Effectors and `[[actuator_node]]` are mutually exclusive (D-029).** D-029
  settles this in prose but the compiler never checked it; a manifest declaring
  both compiled. Now a whole-manifest structural guard,
  `ValidateError::ActuationDoublyDeclared`.

Open questions, recorded as manifest-schema.md v0.7 items 5 and 6. Both are
blob-format changes, so they should ride a bump that is happening anyway rather
than force one. Non-normative sketches below; nothing is decided.

**Q5, variant fields nested vs flat.** Four discriminants gate sibling fields,
and the nesting is inconsistent (`[sensor.nmea0183]`/`[effector.pwm]` nest;
effector geometry, Cyphal sensor `node_id`/`subject`, and bus-kind fields sit
flat). Sketch of the uniform-nesting option, on the rudderboat's bus and
effector:

```toml
# BUS: kind-specific fields move into [bus.<kind>], leaving id/kind/port flat.
[[bus]]
id   = "gnss_serial"
kind = "nmea0183_uart"
port = "uart7"
[bus.nmea0183_uart]
baud     = 4800
checksum = "required"

# EFFECTOR: geometry under [effector.<kind>], wiring under [effector.output].
# Both variant blocks are then self-locating; a misplaced field is a parse
# error (deny_unknown_fields), not a hand-written compiler check.
[[effector]]
id   = "esc_main"
kind = "fixed_thruster"
bus  = "actuator_bridge"
[effector.fixed_thruster]
pos              = [-1.20, 0.00]     # see Q6
azimuth_rad      = 0.0
max_thrust_fwd_n = 300.0
max_thrust_rev_n = 180.0
[effector.output]                    # arm chosen by bus kind, as today
channel   = 0
us_min    = 1100
us_center = 1500
us_max    = 1900
```

The cost is plainly visible: more table headers, deeper nesting, a taller file
for the same vessel. That is the tradeoff to judge: self-documenting shape and
parse-time enforcement, paid for in verbosity.

**Q6, one frame and one notation.** `lever_arm_m = [x,y,z]` on sensors,
`pos_x_m`/`pos_y_m` scalars on effectors, and `[estimator].origin` (currently
parsed and discarded) defining the frame all three share. Sketch:

```toml
# One section owns the body frame; everything positional references it.
[geometry]
origin = "midship_waterline"   # x fwd, y stbd, z down

[[sensor]]
id      = "gnss_main"
role    = "gnss"
bus     = "gnss_serial"
license = "inner_loop"
pos     = [0.0, 0.0, 0.0]      # was lever_arm_m; same notation as effector pos
```

`origin` stays blocked on the estimator answering what it needs from the frame
(D-022); the notation unification (`pos` everywhere) is independent of that and
could land with Q5.

## Signed-blob trust boundary gets the same pseudo-fuzz treatment as the wire parsers

coxswain-manifest was the one untested trust boundary in the fuzz-harness
sense: the no_std blob reader that runs on the H7 against whatever it finds in
flash, i.e. genuinely untrusted bytes, unlike the four wire parsers which
already had `tests/fuzz.rs`. Added the same deterministic, dependency-free
xorshift64* pseudo-fuzz to `crates/coxswain-manifest/tests/fuzz.rs`: random
byte soup and bit-flip/truncate/rotate/insert mutations of a valid signed blob
fed into `read()`, plus a round trip across a spread of random ed25519 signing
seeds (complementing golden.rs's fixed-seed round trip), plus mutated-TOML
fuzzing of the validator as a cheap add-on (not the primary target: the
operator authors TOML by hand, so it isn't exposed to the reader's adversary).

Confirmed the reader's actual contract by reading blob.rs before writing
assertions: framing, then CRC, then ed25519 signature, then postcard decode,
strictly in that order, so it is verify-then-expose, never expose-then-check.
That makes the interesting assertion "any mutated blob that still verifies
must decode to the exact manifest that was signed" (CRC32 collision and
ed25519 forgery are both practically impossible for a random mutation, so this
is checkable without reaching into the crate's internals). 5,000 iterations
each of the byte-soup and mutated-blob loops, no panics, no manifest ever
differed from the one signed.

One tuning note, not a defect: the random-seed round trip does a keygen, a
sign, and a verify per iteration, and ed25519-dalek's debug-build (unoptimized)
scalar arithmetic costs tens of milliseconds per call, unlike the pure
parsing/decoding the other loops do. At 5,000 iterations that test alone took
138s; cut to 200 iterations (`KEY_ITERATIONS`), which still spans a wide range
of key material and brings the whole file back to ~6s.

No defect found: the reader never panicked, never served a mutated payload
past verification, and the round trip held for every seed tried. Full DoD
(workspace tests, clippy, fmt, thumbv7em no_std gate) green.

## Closed-loop Monte-Carlo fault campaign: the same pseudo-fuzz idiom, one level up

Same xorshift64* idiom as the parser/manifest fuzz harnesses, this time
against the full closed loop (`crates/coxswain-hosted/tests/fault_campaign.rs`)
rather than a single parser boundary: `Core::tick` driven through sim ->
estimator -> supervisor -> guidance, with a randomized timeline per seed of
GNSS/heading dropouts, a two-heading-sensor disagreement (bias toggle),
voltage sag, a small geofence ring, and two claimants of different
manifest-declared priority racing register/request/release/heartbeat-silence/
arm/disarm/setpoint. Committed budget: 40 seeds x 600 ticks (24,000 ticks,
~5.6 s in debug, comfortably under the ~10 s target); an exploratory run
before settling on that (3,000 seeds x 1,500 ticks = 4.5M ticks, release
profile, 7.9 s) found nothing beyond what the committed budget already
catches, so the smaller number is not leaving an obvious defect on the table.

Seven invariants asserted every tick, each traced to the exact line in
coxswain-supervisor/coxswain-estimator rather than a guessed contract: no
silent conn yield (conn only changes via `request_conn`, `release_conn`, or
`tick`'s heartbeat-staleness check), estimator state finiteness (the Phase-6
divergence-to-NaN bug class, diary 2026-07-10), health-honesty backstop,
disarmed-never-actuates, critical-voltage-forces-disarm, position-degraded-
idles, and failsafe reachability (health.level == Fault is immediate,
sustained gnss_stale trips within position_degraded_after).

Two harness bugs surfaced and got fixed before the campaign was trustworthy,
both worth recording since they were nearly mistaken for product bugs:

1. **Trusting the `failsafe` field to attribute a conn change is wrong.** A
   heartbeat-staleness revocation (ClaimantLost) can happen on the same tick
   a higher-priority condition (PositionDegraded) is already latched, and the
   matrix's priority order reports the higher one, silently masking the
   actual cause of the conn transition. Fixed by checking the heartbeat-
   staleness path against the harness's own independently-tracked
   `last_heartbeat` timestamps (mirroring `Claimant::last_heartbeat`) instead
   of the `failsafe` field.
2. **Two claimant calls in one tick collapse into one net conn transition
   invariant 1 can't parse.** Letting both a `request_conn` (preemption) and
   a `release_conn` fire for different claimants in the same tick, before
   `Core::tick` runs, produces a Held(A) -> Held(B) -> Unheld sequence that
   only shows up as Held(A) -> Unheld across the tick boundary, with no
   single call explaining it. Capped at one conn-mutating call per tick
   (`odds::CONN_ACTION`, picked uniformly among the candidates) rather than
   trying to model compound transitions.
3. **Timestamp consistency**: claimant calls are stamped with the pre-step
   `now` (mirroring real heartbeats arriving before the tick evaluates them,
   the same convention `closed_loop.rs` already uses), but `sim.step` then
   advances the clock before `Core::tick` runs. Any staleness math in the
   test must compare against the post-step timestamp actually passed to
   `tick()`, not the pre-step one, or every duration is short by one tick.

No supervisor or estimator defect found after these fixes, across both the
committed budget and the deeper exploratory run: authority stayed unique and
attributable, the estimator state stayed finite throughout every fault
interleaving tried (no reproduction of the Phase-6 NaN class), and the
failsafe matrix reached its declared degraded behavior within the declared
timeout in every case. Full DoD (workspace tests, clippy, fmt, thumbv7em
no_std gate) green; the new test is host-only (coxswain-hosted's existing
closed-loop harness, not a core crate) and touches no no_std code.

## Invariant 5 turned into an executed check: host and H7-target run the same estimator/guidance/supervisor byte-for-byte, on this scenario

Until today, "the Linux-hosted profile and the H7/Embassy profile run
byte-for-byte the same estimator, guidance, and supervisor logic" (CLAUDE.md
invariant 5) was verified only by *linking* the core crates for
thumbv7em-none-eabihf (the CI gate). Nothing ever *ran* the target code. Built
the harness to actually run it, under QEMU.

**Design.** Three new crates:

- `coxswain-xprofile-scenario` (workspace member, `no_std`, not in the no_std
  gate's `-p` list): a fixed, deterministic scenario linking the real
  `coxswain-estimator`, `coxswain-guidance`, and `coxswain-supervisor`. 300
  ticks at 10 Hz (30 s): a closed-form truth trajectory (10 s straight leg,
  then a steady turn), GNSS/heading/yaw-rate sampled at 1/2/5 Hz with a
  hand-rolled xorshift64* + Box-Muller RNG (same construction as
  coxswain-estimator's own replay harness, reimplemented against `libm` so it
  is `no_std`), fed through `Estimator::handle`/`state`/`health`, then
  `Supervisor::tick` (autonomy arms itself once healthy, holds a
  `HeadingSpeed` setpoint) and `Guidance::tick`, with the resulting force
  demand fed back as the next tick's actuator command. Every tick emits a
  `Record`: full state (position, heading, body velocity, the 6x6 covariance),
  health (stds, staleness, level, fix mode), and every supervisor/guidance
  *decision* (arming, conn holder, failsafe cause, low-voltage/power-stale
  flags, setpoint kind and payload, force demand) as a fixed sequence of
  canonical `u64`s (`f64::to_bits()` for floats).
- `coxswain-xprofile-target` (deliberately **excluded** from the root
  workspace, `[workspace] exclude` in the root Cargo.toml, own Cargo.lock):
  `#![no_std] #![no_main]` binaries (`cortex-m-rt` entry, `cortex-m-semihosting`
  for stdout, `panic-semihosting` with the `exit` feature so a panic exits
  QEMU nonzero instead of hanging) that link the scenario crate and print
  every record as a `TICK <i> <hex...>` line, then `DONE`. A second binary,
  `blob_overflow_check`, exercises the manifest blob-reader finding (below).
  Excluding this crate from the workspace was necessary, not just tidy: the
  root `.cargo/config.toml` stages `linker = "flip-link"` for the real H7
  firmware phase, and flip-link is not installed in this environment; the
  crate's own `build.rs` supplies the one link argument it needs
  (`-Tlink.x`) directly, and the host test overrides `RUSTFLAGS=""` when
  building it, sidestepping the staged config entirely. Discovered
  empirically that `CARGO_TARGET_<triple>_RUSTFLAGS` does **not** override a
  `.cargo/config.toml` target-specific `rustflags` array (tested in
  isolation); the plain `RUSTFLAGS` env var does, and is what the harness
  uses.
- `coxswain-xprofile-check` (workspace member, host-only): `tests/
  xprofile.rs` runs the scenario natively on host, cross-builds
  `coxswain-xprofile-target`'s binaries for `thumbv7em-none-eabihf`
  (`RUSTFLAGS=""` as above), runs them under QEMU, parses the printed
  records, and diffs host against target field-by-field. Skips cleanly (not
  a failure) when `qemu-system-arm` is not on PATH, exactly mirroring
  `coxswain-hosted/tests/can_rig.rs`'s `require_vcan` runtime-probe pattern.
  Verified the skip path actually triggers (PATH manipulated to exclude
  qemu) and, separately, that the comparison actually detects a divergence
  (flipped one bit of one host record, confirmed the test fails with a
  precise tick/field/ULP report, then reverted).

**QEMU invocation:** `qemu-system-arm -cpu cortex-m7 -machine mps2-an500
-semihosting-config enable=on,target=native -nographic -kernel <elf>`. The
MPS2 AN500 model (Cortex-M7 + FPU) with a generous `memory.x` (4 MB FLASH at
0x0, 4 MB RAM at 0x2000_0000, sized for a throwaway harness binary, not a
real board budget) booted first try; no iteration needed on the memory map.

**Result: MATCH.** All 300 ticks x 57 fields (17,100 field comparisons,
covering position, heading, body velocity, the full 6x6 covariance, every
health flag, and every supervisor/guidance decision) are bit-for-bit
identical between the x86-64 host run and the Cortex-M7 QEMU run, for this
scenario. No ULP-level drift anywhere, including through the hydrodynamic
Fossen prior's trig-heavy Jacobians and the turn segment. The likely reason:
every trig/transcendental call in the estimator, guidance, and model crates
goes through the `libm` crate explicitly (`libm::sin`/`cos`/`atan2`/`sqrt`/
`hypot`, never `f64::sin()` et al.), so both profiles run the identical
portable-Rust software implementation rather than host system libm vs.
target soft/hard-float libm; nalgebra's small fixed-size `SMatrix<f64, 6, 6>`
operations are plain scalar loops at this size, not autovectorized in a way
that would introduce FMA contraction or reordering on one target and not the
other. This is an empirical result for one scenario, not a proof for all
inputs; the doctrine going in was to report a divergence honestly rather than
loosen the comparison, and none showed up to report. No DECISIONS.md change
proposed: invariant 5 needed no reinterpretation, since the strong
(bit-identical) reading held.

**blob.rs finding: confirmed and fixed.** The framing arithmetic in
`crates/coxswain-manifest/src/blob.rs::read` (`HEADER_LEN + payload_len`,
etc., plain `usize` addition on an attacker-controlled `u32` off the wire)
does not overflow on the hosted profile (`usize` is 64-bit; any `u32` fits),
but panics on the actual 32-bit target: `blob_overflow_check` crafted a
10-byte header (valid magic + schema version, `payload_len = u32::MAX`, no
payload/CRC/signature bytes) and ran it through the real `read()` under QEMU.
Before the fix: `panicked at .../blob.rs:76:18: attempt to add with
overflow`, QEMU exit 1. This is exactly the finding flagged going into this
task, now confirmed on real target arithmetic rather than argued from
inspection. Fixed with `checked_add` at each framing step, folding any
overflow into the same `ReadError::Truncated` a merely-short buffer already
gets (a length that overflows the framing math is, definitionally, a length
no real blob can satisfy). Re-ran under QEMU: `NO_PANIC result=Err(Truncated)`,
exit 0. Added a host-side regression test
(`coxswain-manifest/tests/golden.rs::huge_declared_payload_len_is_rejected_not_overflowed`,
passes trivially on 64-bit `usize` but guards the fix from being reverted)
and a target-side one
(`coxswain-xprofile-check::blob_overflow_is_rejected_not_panicked_on_thumbv7em_under_qemu`,
the one that actually exercises the 32-bit-`usize` failure mode). Full
manifest test suite (58 tests across golden.rs and rudderboat.rs) still
green after the fix.

**Wiring.** `.devcontainer/postCreate.sh` now installs `qemu-system-arm`
(idempotent, same style as the zenohd provisioning). `.github/workflows/ci.yml`
gained two new steps (`cargo fmt --check` and `RUSTFLAGS="" cargo clippy
--target thumbv7em-none-eabihf --bins -- -D warnings`, both
`working-directory: crates/coxswain-xprofile-target`, needed because that
crate is outside the workspace and so untouched by the existing
workspace-wide fmt/clippy steps) plus a comment on the existing
`postCreate.sh`/`cargo test --workspace` steps noting they now also
provision and execute the QEMU cross-profile check rather than skip it. The
existing thumbv7em no_std gate (linking coxswain-contract through
coxswain-drivers) is unchanged; `coxswain-xprofile-scenario` is deliberately
not added to it, per the task brief.

New deps, all confined to `coxswain-xprofile-target` (its own Cargo.lock,
never the root workspace's): `cortex-m`, `cortex-m-rt` (minimal reset/vector-
table runtime, the standard way to get a real linked ARM executable rather
than a link-only rlib), `cortex-m-semihosting` (stdout under QEMU; no UART
wired up in this harness), `panic-semihosting` with feature `exit` (a target
panic becomes a printed message plus a nonzero QEMU exit code, so
`blob_overflow_check`'s pre-fix crash was directly observable as a process
exit status, not a hang).

DoD: `cargo test --workspace` green (the two new tests genuinely run QEMU
here, not skip); `cargo clippy --workspace --all-targets -- -D warnings`
green, plus the target crate's own `RUSTFLAGS="" cargo clippy --target
thumbv7em-none-eabihf --bins -- -D warnings` green; `cargo fmt --all --check`
green at the root, plus `cargo fmt --check` green inside the excluded target
crate; the thumbv7em no_std gate's existing `-p` list still builds and is
unchanged.

## Manifest authoring reshape landed (D-030, schema_version 5 -> 6)

Implemented the uniform-nesting reshape sketched earlier today, per D-030. The
authored shape changes; the compiled blob does not (only the `schema_version`
integer). No contract or `types.rs` field renamed, so the blob-format claim held
under `cargo test` (the golden/roundtrip fixtures re-sign and read back
identically after the fixture edits).

Final field-placement map as implemented:

- Bus: `id`/`kind`/`port` flat; kind-gated fields in `[bus.<kind>]`.
  `[bus.cyphal_can]` = `bitrate` + `node_id`; `[bus.nmea2000_can]` = `bitrate` +
  `mode`; `[bus.nmea0183_uart]` = `baud` + `checksum`; `[bus.nmea0183_udp]` =
  `listen_port` + `source_ip` + `segment` + `checksum`; `[bus.uart]` /
  `[bus.actuator_uart]` / `[bus.crsf_uart]` = `baud` (one shared struct).
  `spi`/`i2c`/`pwm` gate nothing, no sub-table.
- Sensor: `id`/`role`/`driver`/`bus`/`license` flat, plus `pos` flat (was
  `lever_arm_m`, still `[f64; 3]`). Role physics under `[sensor.<role>]`:
  `[sensor.gnss].pps`, `[sensor.imu].orientation`,
  `[sensor.compass].{declination_source, declination_deg}`. Transport under the
  bus-kind-selected table: `[sensor.nmea0183]`/`[sensor.nmea2000]` unchanged, new
  `[sensor.cyphal] = {node_id, subject}`.
- Effector: `id`/`kind`/`bus` flat; geometry under `[effector.<kind>]` with
  `pos` at model arity (`[f64; 2]` thruster, `[f64; 1]` rudder, serde
  arity-checked); wiring under `[effector.output]` (required field) with
  `[effector.output.pwm]` for the serial arm.

Enforcement (D-030's real point, ruling C). Nesting + `deny_unknown_fields`
catches a stray field inside a correctly-selected sub-table at parse time, but
not a whole sub-table authored for the wrong discriminant, so the compiler gained

## Lever-arm compensation, increment 3: GNSS position update extended (D-031)

Third of four, the one with real numerical risk: the antenna offset now
feeds a psi column into `update_position`/`update_position_cov`, which is
exactly the coupling that risked reopening the diary/2026-07-10 divergence.

Model (`crates/coxswain-estimator/src/ekf.rs`): for offset `r = [rx, ry]`,
`h_n(x) = n + cos(psi) rx - sin(psi) ry`, `h_e(x) = e + sin(psi) rx +
cos(psi) ry`, matching coxswain-sim's `antenna_position` (increment 1)
exactly. Jacobian nonzero columns n, e, psi: `dh_n/dn = 1, dh_n/dpsi = -sin
rx - cos ry`; `dh_e/de = 1, dh_e/dpsi = cos rx - sin ry`. `update_position`
keeps its two-sequential-scalar-update shape, now routed through the
existing general-Jacobian helper (`scalar_update_h`, already used by
SOG/COG) instead of the direct-index `scalar_update`, since h is no longer a
standard basis vector once psi couples in; each axis's Jacobian is evaluated
at the filter's current psi, so the second (e) update sees the small psi
correction the first (n) update already made. `update_position_cov` keeps
its closed-form two-column Kalman gain; P symmetry still gives `H P = [(P
h_n)^T; (P h_e)^T]`, so the old fast-path shape survives with `P`'s raw
columns 0/1 replaced by `P h_n`/`P h_e`. At `lever_arm_m = [0, 0]` every psi
term is a signed zero and both updates reduce to the pre-D-031 arithmetic
exactly. Routing: `coxswain-estimator::lib.rs` looks up the reporting
sensor's `SensorConfig.lever_arm_m` (a new `Estimator::lever_arm` helper) and
passes it to both update calls; had to hoist the lookup above the `&mut
self.filter` match since a method call borrows all of `self` and the
borrow checker won't split it into disjoint fields the way a direct field
access can.

## Lever-arm compensation, increment 4 (final): GNSS SOG/COG extended (D-031)

Last of four. Position (increment 3) folded the antenna offset's rotation
into `h_n/h_e`; SOG/COG fold in its rotation *rate*: the antenna's body
velocity carries the reference point's `omega x r` term even when the
reference point itself is dead in the water.

Model (`crates/coxswain-estimator/src/ekf.rs`): for offset `r = [rx, ry]` and
state yaw rate `r_yaw = x[5]`, the antenna's body velocity is `ua = u -
r_yaw*ry, va = v + r_yaw*rx`. SOG: `h(x) = s_a = sqrt(ua^2 + va^2)`,
`dh/du = ua/s_a, dh/dv = va/s_a, dh/dr_yaw = (-ua*ry + va*rx)/s_a`. COG:
`h(x) = psi + atan2(va, ua)`, `dh/dpsi = 1, dh/du = -va/s_a^2, dh/dv =
ua/s_a^2, dh/dr_yaw = (ua*rx + va*ry)/s_a^2`. At `lever_arm_m = [0, 0]`,
`ua = u, va = v, s_a = s` and both `dh/dr_yaw` terms vanish: exactly the
pre-D-031 h/Jacobian. Routing is the increment-3 pattern reused verbatim:
`coxswain-estimator::lib.rs` looks up the reporting sensor's `lever_arm_m`
via the existing `Estimator::lever_arm` and passes it to both
`update_sog`/`update_cog`.

Low-speed floor: today's guard tested the reference point's `hypot(u, v)`
against `COG_MIN_SPEED_MPS`. That is the wrong quantity once an offset is
declared — it is the *antenna* speed `s_a` that a receiver measures and
gates on, and `s_a` that sits in the Jacobian denominators above. Both
guards (the `ekf.rs` update-level backstop and the load-bearing intake-level
check in `lib.rs`) now test `s_a`, folding the sensor's `lever_arm_m` and the
filter's yaw rate in before comparing to the floor; at zero offset `s_a =
hypot(u, v)`, unchanged. Confirmed the sim side already agreed (increment 1):
`coxswain-sim`'s `Sim::sample` gates COG on `sampled_sog`, itself computed
from the antenna's `ua, va` — the two sides were already consistent, this
increment just brought the estimator's own guard in line with what the sim
(and a real receiver) actually measures.

Verification:

- **Analytic Jacobian vs. finite difference** (in-repo):
  `sog_cog_offset_jacobian_matches_finite_difference` in `ekf.rs`, four
  non-trivial `(u, v, r, rx, ry)` points, central differences in u, v, r
  (both SOG and COG) and psi (COG), eps = 1e-6. One case initially landed
  exactly on `atan2`'s branch cut (`ua < 0, va = 0`), which blew up the
  central difference in v to ~pi/eps — not a Jacobian bug, a bad choice of
  test point; moved it off the cut and all terms matched analytic to well
  under 1e-6 (the FD truncation floor at this eps).
- **Backward compatibility**: `sog_cog_zero_offset_matches_pre_offset_update`
  (bit-identical `update_sog`/`update_cog` output at `lever_arm_m = [0, 0]`)
  plus the full pre-existing suite (41 estimator unit tests, all built around
  zero offset) passing unchanged.
- **Low-speed-floor interaction**: two targeted checks, one at the `ekf.rs`
  update level and one at the `lib.rs` intake level
  (`sog_and_cog_speed_floor_judges_antenna_speed_not_reference_point_speed`,
  `low_speed_guard_judges_antenna_speed_not_reference_point_speed`). Both
  directions: reference point dead in the water but yawing with a
  side-mounted antenna (`hypot(u,v) = 0`, `s_a = |r*ry| = 0.9` above the
  floor) must fuse, not reject; reference point above the floor
  (`hypot(u,v) = 0.6`) but the omega x r term cancelling it exactly at the
  antenna (`s_a = 0`) must reject, not fuse. An old `hypot(u, v)` guard would
  get both backwards.
- **Closed-loop "prove it matters" replay case** (new, `replay_cases.rs`
  scenario 13): bow offset `[3.0, 0.0]` (same as scenario 12) under a faster
  sustained yaw (15 deg/s, vs. scenario 12's 4 deg/s tuned for the position
  path) so `r*rx = 0.785 m/s` is 39% of the 2 m/s CO speed. One antenna, one
  physical offset — position, SOG, and COG all sampled at the same offset via
  new harness samplers `sample_sog_with_offset`/`sample_cog_with_offset`
  (mirroring `sample_gnss_with_offset`), SOG/COG streams starting after the
  15 s velocity-convergence window (same convention as the existing SOG/COG
  scenarios). Same stream fed to two estimators differing only in declared
  `lever_arm_m`:
  - Told the true offset: surge RMSE 0.10 m/s (CV) / 0.09 m/s (hydro), sway
    RMSE 0.09 m/s (CV) / 0.02 m/s (hydro), mean NEES 1.95 / 1.78 — recovers
    the reference-point body velocity within an honest bound and NEES band.
  - Told `[0, 0]`: sway RMSE 0.80 m/s (CV) / 0.16 m/s (hydro), 6.6-9.1x worse
    than the compensated run on the identical stream. Under constant velocity
    the mean sway bias (0.792 m/s) lands almost exactly on the kinematic
    `r*rx = 0.785 m/s` the naive Jacobian misreads as real sway. Under the
    hydrodynamic prior the bias is markedly smaller (mean 0.125 m/s): the
    commanded `tau` balances the *true* nu, so the process model's own
    dynamics resist the SOG/COG-induced distortion rather than adopting it
    outright. Both models still show compensation mattering by a wide
    margin; the absolute bias floor in the test is set an order of magnitude
    under `r*rx` (not half of it, as first tried) so it holds for both
    without asserting a magnitude the hydrodynamic prior does not actually
    produce — a real, reportable modeling effect, not a test bug.

No negative result on the core claim: compensation recovers the body
velocity, and the uncompensated bias appears in both process models, at a
magnitude that itself depends on how strongly the process model constrains
velocity independently of SOG/COG.

## Q1 settled: estimator fusion is unordered inverse-variance weighting (D-032)

Schema open question 1 was three tangled strands. Two were already answered by
the estimator and only needed the schema to say so; one stays parked for a
stated reason.

- Priority-vs-weights: the `gnss`/`heading`/`imu` lists never used order. The
  estimator fuses every licensed sensor in a set as it arrives, inverse-variance
  weighted by declared std. Blessed as the fusion contract (D-032); failover
  tiers deferred until a vessel carries redundant sensors of very different
  quality. The "fusion priority order; provisional" example comment was simply
  wrong and is fixed.
- Separate velocity list: SOG/COG ride the `gnss` set (same physical receiver);
  no separate list until a vessel has an independent velocity source.
- Per-vessel noise parameters: stays parked on the system-ID campaign. Q is
  fixed per-model in code, R is the per-measurement declared std; authoring
  per-vessel values needs identification data that does not exist, and growing
  schema fields that cannot be populated is the D-022 anti-pattern. This is now
  the whole of schema open question 1 (renumbered under v0.8).

No blob or schema_version change: the lists stay arrays of sensor ids. Added one
validation rule, `EstimatorSensorDuplicated` (a set has no repeats, and a repeat
would double-count in the fusion), with a golden rejection test. De-provisionalized
the two estimator source comments that pointed at "open question 1".
