# 2026-07-14

## Cyphal actuator backend: increment 2c plan (handoff)

Increments 1, 2a, 2b are done and pushed on branch `cyphal-actuator-backend`
(coxswain-cyphal transport crate; the OutputBackend trait + $CXOUT refactor to
the physical-units boundary, D-028; the CyphalActuatorBackend with
command-then-report). 2c is the hosted integration, one coupled unit, to be
done as its own focused session. Four parts:

1. **Effector schema (signed blob, schema_version 4 -> 5).** An effector's
   output wiring depends on its bus kind. Cleanest shape: an `EffectorOutput`
   enum, `Serial { channel, pwm }` vs `Cyphal { node_id, command_subject,
   feedback_subject }`, selected by the effector's bus kind at compile.
   Validation: a `cyphal_can` effector requires the Cyphal fields (not
   channel/pwm); an `actuator_uart`/`pwm` effector requires channel/pwm (not
   subjects). Blast radius: bump schema_version, update every golden/tamper/
   version-rejection test, the manifest schema doc, and the blob reader; add a
   `cyphal_can` effector to a test manifest. Record as a DECISIONS entry
   (D-029, or a D-028 follow-on) BEFORE coding, since it changes the blob
   format (D-018).

2. **Read-write CAN transport (`hosted/src/can.rs`).** Today it is listen-only
   by construction (N2K, D-011), with an explicit "no write path" invariant.
   The Cyphal control bus is D-011's transmit-allowed exception: add a
   `write_frame` and a read-write open, and update the module doc to carve out
   that exception (keep the listen-only path for N2K).

3. **Hosted wiring.** For a `cyphal_can` bus carrying effectors, build a
   `CyphalActuatorBackend` from the manifest (per-effector CyphalEffector
   subjects, conn node id, power subject, tolerance), select it as that bus's
   output backend, drive it through the trait with a CanSink writing frames,
   and read feedback via `can::spawn_reader` -> `handle_frame` -> `core.power`
   (reuse the existing PowerStatus intake) plus the command-then-report
   divergence into supervisor health.

4. **vcan rig (`hosted/tests/cyphal_can_rig.rs`), mirroring can_rig.rs.** The
   test plays the actuator node: reads command frames off vcan, sends feedback
   frames back, and asserts the command-then-report result and power reach the
   observable surface (health telemetry). Napkin-scale byte tests run
   everywhere; the vcan end-to-end test skips gracefully where vcan is
   unavailable (the require_vcan pattern).

Kick-off for next session: check out `cyphal-actuator-backend`, then start with
part 1 (the schema DECISIONS entry, then the schema_version bump), verifying at
each step (golden tests, no_std gate). Eventually merge to main directly (no
PR), per the operator.

## Cyphal actuator backend: increment 2c done

2c landed. The three open schema decisions from the handoff went, after
discussion with the operator: (1) the conn node's own bus id lives as `node_id`
on the `cyphal_can` `[[bus]]`, and the power node's voltage subject as `subject`
on the role=power `[[sensor]]`; (2) `report_tolerance` is per effector, in that
effector's physical units (a single bus scalar cannot be right for both a
thruster in newtons and a rudder in radians); (3) effectors and
`[[actuator_node]]` stay mutually exclusive, as on the serial path. All three
are recorded in D-029, written before the code.

What shipped:

- Schema v4 -> v5. `EffectorEntry.output` is now an `EffectorOutput` sum type,
  `Serial { channel, pwm }` vs `Cyphal { node_id, command_subject,
  feedback_subject, report_tolerance }`, selected by the effector's bus kind at
  compile; the wrong arm's fields are rejected. `BusEntry.node_id` and
  `SensorEntry.subject` added. coxswain-manifest now depends on coxswain-cyphal
  for the node-id/subject-id wire ranges (single source of truth). New
  `cyphal_effector.rs` golden + one rejection per new rule.
- Backend refactor: `report_tolerance` moved onto `CyphalEffector`;
  `power_subject` is now `Option` (the failsafe matrix already tolerates no
  power link, so a Cyphal actuator bus need not have a power node).
- `hosted/src/can.rs` gained `write_frame`; the module's listen-only invariant
  is carved to the N2K path, with the Cyphal control bus as D-011's
  transmit-allowed exception.
- Hosted wiring: a `CyphalActuatorLink` per `cyphal_can` bus carrying
  effectors, built from the manifest, driven through the output backend trait
  with a `CanSink`, feedback/power read via `can::spawn_reader` ->
  `handle_frame`. Power feeds the existing `PowerStatus` intake; per-effector
  divergence and the supervisor's report-only low-voltage flag surface in the
  published health as `actuation` and supervisor `power` subjects. A new
  `ClaimantClient::subscribe_health` distills both, so a claimant observes them
  without arming.
- vcan rig `hosted/tests/cyphal_can_rig.rs`, mirroring can_rig.rs: byte-level
  napkin tests run everywhere; the end-to-end test plays the actuator/power
  node and asserts command frames, divergence, and voltage reach health, and
  skips gracefully where vcan is unavailable. A no-vcan boot-error test drives
  the real binary through the whole Cyphal boot path (an unopenable control bus
  fails boot).

Not done, deferred honestly: node-local failsafe/heartbeat for a Cyphal
effector (the `[[actuator_node]]` fields) waits for the actuator firmware that
would enforce it (Phase 9); the sensor-only Cyphal read path (a power node on a
bus with no effectors) is not wired, since nothing needs it yet.

## Schema UX review: two enforcement gaps closed, two open questions raised

Went over the manifest TOML as the single point of contact with the
commissioning engineer, judging it for coherence rather than correctness. The
spine holds up (identity -> board profile -> buses -> things that reference
buses -> policy; string-id references; `license` as the one trust bit; unit
suffixes; defaults for plumbing but never for trust or physics). Two settled
invariants turned out not to be enforced, and two genuine consistency questions
surfaced that want deciding before the next `schema_version` bump.

Enforcement gaps closed now (both enforce already-settled decisions, no schema
change, no version bump):

- **`[rc].claimant` must name a declared `[[claimant]]` (D-025).** The RC's
  preemption priority lives in `[[claimant]]`, keyed by that integer. A `[rc]`
  pointing at an id no claimant declares compiled cleanly and the RC silently
  ran at the default priority 0, the opposite of "the hand controller outranks
  autonomy". Now a `ValidateError::RcClaimantUnknown`.
- **Effectors and `[[actuator_node]]` are mutually exclusive (D-029).** D-029
  settles this in prose but the compiler never checked it; a manifest declaring
  both compiled. Now a whole-manifest structural guard,
  `ValidateError::ActuationDoublyDeclared`.

Open questions, recorded as manifest-schema.md v0.7 items 5 and 6. Both are
blob-format changes, so they should ride a bump that is happening anyway rather
than force one. Non-normative sketches below; nothing is decided.

**Q5, variant fields nested vs flat.** Four discriminants gate sibling fields,
and the nesting is inconsistent (`[sensor.nmea0183]`/`[effector.pwm]` nest;
effector geometry, Cyphal sensor `node_id`/`subject`, and bus-kind fields sit
flat). Sketch of the uniform-nesting option, on the rudderboat's bus and
effector:

```toml
# BUS: kind-specific fields move into [bus.<kind>], leaving id/kind/port flat.
[[bus]]
id   = "gnss_serial"
kind = "nmea0183_uart"
port = "uart7"
[bus.nmea0183_uart]
baud     = 4800
checksum = "required"

# EFFECTOR: geometry under [effector.<kind>], wiring under [effector.output].
# Both variant blocks are then self-locating; a misplaced field is a parse
# error (deny_unknown_fields), not a hand-written compiler check.
[[effector]]
id   = "esc_main"
kind = "fixed_thruster"
bus  = "actuator_bridge"
[effector.fixed_thruster]
pos              = [-1.20, 0.00]     # see Q6
azimuth_rad      = 0.0
max_thrust_fwd_n = 300.0
max_thrust_rev_n = 180.0
[effector.output]                    # arm chosen by bus kind, as today
channel   = 0
us_min    = 1100
us_center = 1500
us_max    = 1900
```

The cost is plainly visible: more table headers, deeper nesting, a taller file
for the same vessel. That is the tradeoff to judge: self-documenting shape and
parse-time enforcement, paid for in verbosity.

**Q6, one frame and one notation.** `lever_arm_m = [x,y,z]` on sensors,
`pos_x_m`/`pos_y_m` scalars on effectors, and `[estimator].origin` (currently
parsed and discarded) defining the frame all three share. Sketch:

```toml
# One section owns the body frame; everything positional references it.
[geometry]
origin = "midship_waterline"   # x fwd, y stbd, z down

[[sensor]]
id      = "gnss_main"
role    = "gnss"
bus     = "gnss_serial"
license = "inner_loop"
pos     = [0.0, 0.0, 0.0]      # was lever_arm_m; same notation as effector pos
```

`origin` stays blocked on the estimator answering what it needs from the frame
(D-022); the notation unification (`pos` everywhere) is independent of that and
could land with Q5.

## No-allocation invariant now enforced by a host test

CLAUDE.md invariant 5 / D-020 ("no allocation in the control path") was a
convention only; nothing failed CI if it were ever broken. Added
`crates/coxswain-hosted/tests/no_alloc.rs`: a hand-rolled counting global
allocator (wraps `std::alloc::System`, an `AtomicBool` arm/disarm flag plus an
`AtomicUsize` counter, no new dependency) arms only around `Core::ingest`,
`Core::power`, and `Core::tick` for one tick, i.e. the estimator's
predict-and-update, the supervisor's failsafe matrix, guidance, and the
allocator (the effector table is non-empty so the allocation stage actually
runs). `Simulator::step`/`apply_outputs` (the host-only plant, which does use
`Vec`) stay outside the guarded window. Ran it: zero allocations, so the claim
holds today for the estimator/guidance/supervisor/allocation path as actually
composed in `coxswain-hosted::Core::tick`. Full DoD (host tests, clippy,
fmt, thumbv7em gate for the core crates) stays green; the new test is
host-only and touches no no_std code.

## Signed-blob trust boundary gets the same pseudo-fuzz treatment as the wire parsers

coxswain-manifest was the one untested trust boundary in the fuzz-harness
sense: the no_std blob reader that runs on the H7 against whatever it finds in
flash, i.e. genuinely untrusted bytes, unlike the four wire parsers which
already had `tests/fuzz.rs`. Added the same deterministic, dependency-free
xorshift64* pseudo-fuzz to `crates/coxswain-manifest/tests/fuzz.rs`: random
byte soup and bit-flip/truncate/rotate/insert mutations of a valid signed blob
fed into `read()`, plus a round trip across a spread of random ed25519 signing
seeds (complementing golden.rs's fixed-seed round trip), plus mutated-TOML
fuzzing of the validator as a cheap add-on (not the primary target: the
operator authors TOML by hand, so it isn't exposed to the reader's adversary).

Confirmed the reader's actual contract by reading blob.rs before writing
assertions: framing, then CRC, then ed25519 signature, then postcard decode,
strictly in that order, so it is verify-then-expose, never expose-then-check.
That makes the interesting assertion "any mutated blob that still verifies
must decode to the exact manifest that was signed" (CRC32 collision and
ed25519 forgery are both practically impossible for a random mutation, so this
is checkable without reaching into the crate's internals). 5,000 iterations
each of the byte-soup and mutated-blob loops, no panics, no manifest ever
differed from the one signed.

One tuning note, not a defect: the random-seed round trip does a keygen, a
sign, and a verify per iteration, and ed25519-dalek's debug-build (unoptimized)
scalar arithmetic costs tens of milliseconds per call, unlike the pure
parsing/decoding the other loops do. At 5,000 iterations that test alone took
138s; cut to 200 iterations (`KEY_ITERATIONS`), which still spans a wide range
of key material and brings the whole file back to ~6s.

No defect found: the reader never panicked, never served a mutated payload
past verification, and the round trip held for every seed tried. Full DoD
(workspace tests, clippy, fmt, thumbv7em no_std gate) green.

## Closed-loop Monte-Carlo fault campaign: the same pseudo-fuzz idiom, one level up

Same xorshift64* idiom as the parser/manifest fuzz harnesses, this time
against the full closed loop (`crates/coxswain-hosted/tests/fault_campaign.rs`)
rather than a single parser boundary: `Core::tick` driven through sim ->
estimator -> supervisor -> guidance, with a randomized timeline per seed of
GNSS/heading dropouts, a two-heading-sensor disagreement (bias toggle),
voltage sag, a small geofence ring, and two claimants of different
manifest-declared priority racing register/request/release/heartbeat-silence/
arm/disarm/setpoint. Committed budget: 40 seeds x 600 ticks (24,000 ticks,
~5.6 s in debug, comfortably under the ~10 s target); an exploratory run
before settling on that (3,000 seeds x 1,500 ticks = 4.5M ticks, release
profile, 7.9 s) found nothing beyond what the committed budget already
catches, so the smaller number is not leaving an obvious defect on the table.

Seven invariants asserted every tick, each traced to the exact line in
coxswain-supervisor/coxswain-estimator rather than a guessed contract: no
silent conn yield (conn only changes via `request_conn`, `release_conn`, or
`tick`'s heartbeat-staleness check), estimator state finiteness (the Phase-6
divergence-to-NaN bug class, diary 2026-07-10), health-honesty backstop,
disarmed-never-actuates, critical-voltage-forces-disarm, position-degraded-
idles, and failsafe reachability (health.level == Fault is immediate,
sustained gnss_stale trips within position_degraded_after).

Two harness bugs surfaced and got fixed before the campaign was trustworthy,
both worth recording since they were nearly mistaken for product bugs:

1. **Trusting the `failsafe` field to attribute a conn change is wrong.** A
   heartbeat-staleness revocation (ClaimantLost) can happen on the same tick
   a higher-priority condition (PositionDegraded) is already latched, and the
   matrix's priority order reports the higher one, silently masking the
   actual cause of the conn transition. Fixed by checking the heartbeat-
   staleness path against the harness's own independently-tracked
   `last_heartbeat` timestamps (mirroring `Claimant::last_heartbeat`) instead
   of the `failsafe` field.
2. **Two claimant calls in one tick collapse into one net conn transition
   invariant 1 can't parse.** Letting both a `request_conn` (preemption) and
   a `release_conn` fire for different claimants in the same tick, before
   `Core::tick` runs, produces a Held(A) -> Held(B) -> Unheld sequence that
   only shows up as Held(A) -> Unheld across the tick boundary, with no
   single call explaining it. Capped at one conn-mutating call per tick
   (`odds::CONN_ACTION`, picked uniformly among the candidates) rather than
   trying to model compound transitions.
3. **Timestamp consistency**: claimant calls are stamped with the pre-step
   `now` (mirroring real heartbeats arriving before the tick evaluates them,
   the same convention `closed_loop.rs` already uses), but `sim.step` then
   advances the clock before `Core::tick` runs. Any staleness math in the
   test must compare against the post-step timestamp actually passed to
   `tick()`, not the pre-step one, or every duration is short by one tick.

No supervisor or estimator defect found after these fixes, across both the
committed budget and the deeper exploratory run: authority stayed unique and
attributable, the estimator state stayed finite throughout every fault
interleaving tried (no reproduction of the Phase-6 NaN class), and the
failsafe matrix reached its declared degraded behavior within the declared
timeout in every case. Full DoD (workspace tests, clippy, fmt, thumbv7em
no_std gate) green; the new test is host-only (coxswain-hosted's existing
closed-loop harness, not a core crate) and touches no no_std code.
