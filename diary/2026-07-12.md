# 2026-07-12

## Power report staleness closed in the failsafe matrix

The open item from 2026-07-11: if `$CXPWR` reports stop arriving, the
supervisor kept trusting the last voltage forever, so both voltage
failsafes silently rotted. Settled consistent with matrix v1's philosophy
(loss of measurement is not loss of battery):

`SupervisorConfig` gains `power_stale_after: Duration`. The supervisor
tracks the timestamp of the last *finite* power reading; once one has been
seen, a tick more than `power_stale_after` past it refuses arming
(`ArmError::PowerStale`, mirroring `VoltageLow`) and sets a new
`Directive::power_stale` report-only flag. An armed vessel stays armed and
keeps its setpoint — report-only, exactly like low voltage. Before any
report has ever arrived, the existing healthy-default behavior is
unchanged: the clock only starts on a genuine report, not on the caller's
boot-time placeholder.

That last point needed a companion fix: `coxswain-hosted::Core::new`'s
power sentinel changed from a nominal 13.0 V to `f64::NAN`. With a finite
default, the very first tick would have been indistinguishable from a real
report and the vessel would go falsely stale a few seconds after boot on
any hull with no power link wired up at all — exactly the deployment the
matrix is supposed to keep arm-able. NaN already gets the "no good reading
yet, treat as within bounds" treatment from the existing voltage guard;
extending that guard to also mean "never counts as a report" closed both
problems with one change, and is why a NaN report must not refresh the
staleness clock either (same reasoning, tested directly).

Not yet an authored manifest field: `coxswain-manifest::compile` hardcodes
`Duration::from_millis(3000)` with a comment that the schema bump is a
follow-up. `docs/hardware.md`'s known-gaps note updated to close the item.

One test-only wrinkle: `coxswain-hosted`'s `power_report_rig` desk-rig test
wrote `$CXPWR` as one-shot bytes per stage, which would now go stale
before the estimator/GNSS convergence even finishes. Replaced with a
`PowerTransmitter` (mirrors the existing `RcTransmitter` pattern) that
keeps a report flowing at a safe cadence but writes immediately on a
deliberate voltage change, preserving the original near-instant
arm-refusal latency the test's timing margins depend on.

## Queue closed: range guards, RC as vessel data, replay promoted, N2K reaches a CAN bus

Five more tasks landed behind the staleness entry above, each behind the
full gate. The NaN boot sentinel is that entry's detail and is not
repeated here.

Range validation at the wire and estimator intake, the diary's own
"lat/lon/heading unbounded" marker from 2026-07-11: Keelson conversion
now rejects geodetically impossible setpoint fields, and estimator
intake rejects out-of-range `GnssPosition` and any non-positive declared
std ahead of the Kalman gain. The finding that makes the guard more than
defensive: confirmed end to end that a syntactically valid GGA sentence
carrying latitude 91 deg parses cleanly, passes the fix-quality gate, and
reaches the estimator. Nothing upstream of the intake guard would have
stopped it; the guard is load-bearing.

coxswain-manifest v0.5 (wire schema 3 -> 4) gave RC a `[rc]` section:
claimant id and the full channel mapping the hosted profile hardcoded
until now, compiled to a typed `RcEntry` alongside the effector entries.
The mapping stops being a CLI-adjacent implementation detail and becomes
vessel data under the same CRC + ed25519 signature as the effector
table: a mission's RC channel assignment is now provable against the
manifest hash it ran under. `coxswain-hosted` picks it up the same way
it picks up `actuator_uart`, `--port <bus_id>=<device>`, with symmetric
boot errors for a declared-but-unmapped or mapped-but-undeclared bus.
sd_notify rode along on the same task (READY=1, WATCHDOG=1 at 1 Hz),
process-level, complementing rather than replacing the conn node's
hardware watchdog.

coxswain-replay: the 2026-07-10 promotion condition (a second consumer
of the harness's JSONL format) is met, so the format graduated out of
`coxswain-estimator`'s test harness into its own crate, alongside a
raw-NMEA log format for `--record-nmea`. `cxconvert` bridges the two,
running a raw log through the real `Gnss0183Driver` into measurement
JSONL. The end-to-end proof without hardware: a synthetic GGA+HDT log
through `cxconvert` into a real `Estimator` converges to 4 cm position
error against truth (the test's own bound is a loose < 5 m, same margin
as the desk rig's; the measured figure is the one worth keeping).

coxswain-n2k gained fast-packet reassembly (fixed pool of four 223-byte
slots keyed on source and PGN, new first-frames restart a key, gapped
counters drop the slot) and PGN 129029 GNSS Position Data. `coxswain-hosted`
then wired an `nmea2000_can` bus to raw SocketCAN, listen-only by
construction (never a write, own-message reception off), decoded values
publishing to Keelson as raw enrichment under the sensor's source_id.
Frame-byte round-trip tests independent of vcan caught a real bug before
the vcan rig or CI ever did: a missing J1939 data-page bit (the DP bit
at bit 24 of the 29-bit identifier, `coxswain-n2k/src/id.rs`) in the
test harness's own frame-id construction. Cheaper to catch in a
microsecond-scale unit test than a few seconds into a virtual-interface
round trip.

Environment note, for honesty: vcan does not run in this devcontainer.
The WSL2 kernel has `CONFIG_CAN_VCAN` unset and no `/lib/modules` for
`modprobe`, so `crates/coxswain-hosted/tests/can_rig.rs`'s end-to-end
rig skips here by design, and did through every run of this queue. It
runs in CI behind a best-effort `modprobe vcan` on the Ubuntu runner.
Until a vcan-capable kernel or real CAN hardware shows up locally, the
real CAN path is exercised two ways: unit tests independent of any
interface (the id-decode fix above), and CI's vcan rig, not a run on
this machine.
