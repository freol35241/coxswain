# 2026-07-12

## Power report staleness closed in the failsafe matrix

The open item from 2026-07-11: if `$CXPWR` reports stop arriving, the
supervisor kept trusting the last voltage forever, so both voltage
failsafes silently rotted. Settled consistent with matrix v1's philosophy
(loss of measurement is not loss of battery):

`SupervisorConfig` gains `power_stale_after: Duration`. The supervisor
tracks the timestamp of the last *finite* power reading; once one has been
seen, a tick more than `power_stale_after` past it refuses arming
(`ArmError::PowerStale`, mirroring `VoltageLow`) and sets a new
`Directive::power_stale` report-only flag. An armed vessel stays armed and
keeps its setpoint — report-only, exactly like low voltage. Before any
report has ever arrived, the existing healthy-default behavior is
unchanged: the clock only starts on a genuine report, not on the caller's
boot-time placeholder.

That last point needed a companion fix: `coxswain-hosted::Core::new`'s
power sentinel changed from a nominal 13.0 V to `f64::NAN`. With a finite
default, the very first tick would have been indistinguishable from a real
report and the vessel would go falsely stale a few seconds after boot on
any hull with no power link wired up at all — exactly the deployment the
matrix is supposed to keep arm-able. NaN already gets the "no good reading
yet, treat as within bounds" treatment from the existing voltage guard;
extending that guard to also mean "never counts as a report" closed both
problems with one change, and is why a NaN report must not refresh the
staleness clock either (same reasoning, tested directly).

Not yet an authored manifest field: `coxswain-manifest::compile` hardcodes
`Duration::from_millis(3000)` with a comment that the schema bump is a
follow-up. `docs/hardware.md`'s known-gaps note updated to close the item.

One test-only wrinkle: `coxswain-hosted`'s `power_report_rig` desk-rig test
wrote `$CXPWR` as one-shot bytes per stage, which would now go stale
before the estimator/GNSS convergence even finishes. Replaced with a
`PowerTransmitter` (mirrors the existing `RcTransmitter` pattern) that
keeps a report flowing at a safe cadence but writes immediately on a
deliberate voltage change, preserving the original near-instant
arm-refusal latency the test's timing margins depend on.

## Queue closed: range guards, RC as vessel data, replay promoted, N2K reaches a CAN bus

Five more tasks landed behind the staleness entry above, each behind the
full gate. The NaN boot sentinel is that entry's detail and is not
repeated here.

Range validation at the wire and estimator intake, the diary's own
"lat/lon/heading unbounded" marker from 2026-07-11: Keelson conversion
now rejects geodetically impossible setpoint fields, and estimator
intake rejects out-of-range `GnssPosition` and any non-positive declared
std ahead of the Kalman gain. The finding that makes the guard more than
defensive: confirmed end to end that a syntactically valid GGA sentence
carrying latitude 91 deg parses cleanly, passes the fix-quality gate, and
reaches the estimator. Nothing upstream of the intake guard would have
stopped it; the guard is load-bearing.

coxswain-manifest v0.5 (wire schema 3 -> 4) gave RC a `[rc]` section:
claimant id and the full channel mapping the hosted profile hardcoded
until now, compiled to a typed `RcEntry` alongside the effector entries.
The mapping stops being a CLI-adjacent implementation detail and becomes
vessel data under the same CRC + ed25519 signature as the effector
table: a mission's RC channel assignment is now provable against the
manifest hash it ran under. `coxswain-hosted` picks it up the same way
it picks up `actuator_uart`, `--port <bus_id>=<device>`, with symmetric
boot errors for a declared-but-unmapped or mapped-but-undeclared bus.
sd_notify rode along on the same task (READY=1, WATCHDOG=1 at 1 Hz),
process-level, complementing rather than replacing the conn node's
hardware watchdog.

coxswain-replay: the 2026-07-10 promotion condition (a second consumer
of the harness's JSONL format) is met, so the format graduated out of
`coxswain-estimator`'s test harness into its own crate, alongside a
raw-NMEA log format for `--record-nmea`. `cxconvert` bridges the two,
running a raw log through the real `Gnss0183Driver` into measurement
JSONL. The end-to-end proof without hardware: a synthetic GGA+HDT log
through `cxconvert` into a real `Estimator` converges to 4 cm position
error against truth (the test's own bound is a loose < 5 m, same margin
as the desk rig's; the measured figure is the one worth keeping).

coxswain-n2k gained fast-packet reassembly (fixed pool of four 223-byte
slots keyed on source and PGN, new first-frames restart a key, gapped
counters drop the slot) and PGN 129029 GNSS Position Data. `coxswain-hosted`
then wired an `nmea2000_can` bus to raw SocketCAN, listen-only by
construction (never a write, own-message reception off), decoded values
publishing to Keelson as raw enrichment under the sensor's source_id.
Frame-byte round-trip tests independent of vcan caught a real bug before
the vcan rig or CI ever did: a missing J1939 data-page bit (the DP bit
at bit 24 of the 29-bit identifier, `coxswain-n2k/src/id.rs`) in the
test harness's own frame-id construction. Cheaper to catch in a
microsecond-scale unit test than a few seconds into a virtual-interface
round trip.

Environment note, for honesty: vcan does not run in this devcontainer.
The WSL2 kernel has `CONFIG_CAN_VCAN` unset and no `/lib/modules` for
`modprobe`, so `crates/coxswain-hosted/tests/can_rig.rs`'s end-to-end
rig skips here by design, and did through every run of this queue. It
runs in CI behind a best-effort `modprobe vcan` on the Ubuntu runner.
Until a vcan-capable kernel or real CAN hardware shows up locally, the
real CAN path is exercised two ways: unit tests independent of any
interface (the id-decode fix above), and CI's vcan rig, not a run on
this machine.

## SOG/COG fusion and covariance/RTK intake: the two ears

A vessel's GNSS receiver reports two independent things about motion: a
compass-like heading (where the bow points) and, separately, speed and
course over ground (where the hull is actually going, from Doppler or
successive fixes). The estimator has fused the first since Phase 2 and
ignored the second entirely; RMC/VTG parsed and were discarded
(gnss0183.rs's own comment: "MeasurementKind has no SOG/COG variant").
This closes that gap, plus the parallel gap on the position side: a
scalar HDOP-derived std discards everything an RTK receiver knows about
its own correlated N/E error and fix status. Two ears, not one: heading
tells the filter which way the bow points, COG tells it which way the
hull moves, and they diverge under sway, current, and leeway exactly
where it matters for a small boat with no dynamic positioning.

Settled design points:

- **Polar measurement models, not linear.** SOG measures h(x) =
  sqrt(u^2 + v^2), COG measures h(x) = psi + atan2(v, u). Both are
  genuinely nonlinear in the velocity states, so both get analytic
  Jacobians verified against central finite differences (1e-6 tolerance
  class, `coxswain-estimator/src/ekf.rs`'s test module) rather than the
  sequential-scalar linear-h shortcut the existing GNSS/heading/yaw-rate
  updates use. A new `scalar_update_h` generalizes the existing
  `scalar_update` to an arbitrary Jacobian row instead of a single state
  index; `scalar_update` itself is untouched; the risk of touching code
  every existing replay case depends on wasn't worth the line count
  saved by unifying them.
- **v1 licensing rides the gnss list, no separate velocity list.** SOG
  and COG fuse exactly when their sensor id sits in
  `EstimatorConfig::gnss` with `inner_loop` license, the premise being
  that they come off the same physical receiver as position. Recorded as
  an open point (schema open question 1, D-022) rather than answered:
  a receiver that reports velocity from a different sensor than its
  position fix is a real but currently unhandled case.
- **The 0.5 m/s floor, duplicated on purpose at three seams.** Below
  `COG_MIN_SPEED_MPS`, course over ground is direction-over-noise and
  the COG Jacobian divides by s^2. The constant is defined once in
  coxswain-estimator (`COG_MIN_SPEED_MPS`, re-exported) and copied
  verbatim into `coxswain-drivers::gnss0183` (gates COG emission on the
  sentence's own reported SOG) and `coxswain-sim` (gates the simulated
  velocity sensor's COG on its *sampled* SOG, matching what a real
  receiver would do). Each copy is commented on why it isn't shared: the
  driver and simulator floors are about what a sentence's own numbers
  can support, sentence-local physics; the estimator's is a numerical
  backstop for its own Jacobians. They share a value today, not a
  purpose. The estimator-side guard is checked *before* `predict` runs
  (using the filter's speed estimate as of the last accepted
  measurement), so a `LowSpeed` rejection costs the filter nothing --
  the same "reject before touching state" property every other typed
  rejection has.
- **RMC is the sole SOG/COG source; VTG parses and discards.** VTG
  duplicates RMC's SOG/COG on any receiver that emits both (same track,
  same instant); emitting from both would double-count the exact way a
  position out of RMC would double-count GGA. RMC gates on
  `status == Valid` and the FAA mode (Autonomous/Differential/FloatRtk/
  FixedRtk trusted, Estimated/Manual/Simulator/NotValid not, `None` --
  a pre-2.3 sentence with no mode field -- trusted on status alone,
  since that is the only signal a pre-2.3 receiver gives).
- **`GnssPositionCov` is additive, not a replacement.** The existing
  scalar `GnssPosition` variant is untouched; `GnssPositionCov` carries
  a full 2x2 NE covariance (m^2) and a new `GnssFixMode` enum
  (telemetry vocabulary, not a trust policy -- v1 does not gate fusion
  on it) for a source that actually knows its own correlated error and
  RTK status. The estimator update is closed-form (2x2 innovation
  inverse by hand, not a generic matrix solve) rather than two
  sequential scalar updates, since a real covariance need not be
  diagonal. Intake validates finite, symmetric within 1e-6 relative
  tolerance, and positive-definite (diagonal positivity + determinant);
  failing either reuses `InvalidStd` rather than a new rejection variant
  (both mean the same thing: the declared uncertainty cannot drive the
  Kalman gain). `EstimatorHealth` gains `fix: Option<GnssFixMode>`, the
  mode of the most recently *accepted position* measurement -- a
  `SpeedOverGround`/`CourseOverGround` sample does not touch it, and a
  following plain `GnssPosition` (which carries no mode) clears it back
  to `None`. 0183 stays on the scalar path forever (no covariance on
  that wire); the field simply never gets set from that source, which
  `gnss0183.rs`'s own module doc now says is deliberate.
- **RMC can yield two measurements from one sentence; the driver's
  `push` returns a small `MeasurementBatch` (1 or 2) instead of a single
  `Measurement`.** Every other sentence still yields at most one. This
  is the one real API break in the driver crate; `coxswain-hosted` and
  `coxswain-replay::convert` both updated to iterate the batch.

### THE EXPERIMENT: compass loss with COG as backup, both process models, 1 and 5 Hz

Methodology from the 2026-07-10 yaw-rate degradation entry, reused
directly: seeded, deterministic streams, both process models, straight
line at cruise speed (3 m/s), GNSS 1 Hz/2 m and gyro 20 Hz/0.01 rad/s
held constant throughout. The heading source dies at t = 50 s on a
150 s run and never returns (`compass_loss_with_cog_stays_finite_and_
bounded`, `crates/coxswain-estimator/tests/replay_cases.rs`); SOG/COG,
when present, fuse throughout at 1 Hz or 5 Hz. Errors probed over
t = 60-150 s, well clear of the loss transient.

| process model | COG rate | heading RMSE, no COG | heading RMSE, with COG | mean NEES, with COG |
|---|---|---|---|---|
| constant-velocity | 1 Hz | 1.263 deg | 1.309 deg | 1.52 |
| constant-velocity | 5 Hz | 1.263 deg | 1.293 deg | 1.80 |
| hydrodynamic | 1 Hz | 0.709 deg | 0.467 deg | 1.40 |
| hydrodynamic | 5 Hz | 0.709 deg | 0.380 deg | 1.84 |

The filter stays finite throughout every combination; no divergence,
with or without COG, under this trajectory (a straight line -- gyro
alone dead-reckons heading as a bias-plus-random-walk that a
constant-yaw-rate line does not stress the way a real maneuver would).

The honest finding is a split, not a clean win: under the hydrodynamic
prior COG recovers 34-46% of the heading RMSE the compass loss would
otherwise cost, because the prior's own dynamics keep [u, v] tracking
truth well enough that COG's psi + atan2(v, u) measurement is a real
correction. Under constant velocity, COG does not help and is a small
wash (3-4% worse): with no process model coupling velocity to anything,
[u, v] wander on their own noise budget, and COG's contribution is
dominated by that wander rather than by genuine information about psi.
COG's value as a compass backup is conditional on the process model
actually constraining velocity, not a universal property of fusing it.
Mean NEES stays inside [1.4, 1.8], the same "conservative, not
overconfident" band the rest of the replay suite reports, in every
combination -- the filter does not become overconfident either
holding COG or losing the compass.

Locked as `compass_loss_with_cog_stays_finite_and_bounded`: finite
throughout and heading RMSE under 3 deg (guidance's settle band) for
every (process model, COG rate) combination with COG present; the
without-COG numbers are reported, not asserted on, since "current
behavior" is what is being characterized, not fixed.

### Other regression numbers

Noisy straight line, SOG/COG added at 2 Hz alongside the existing
baseline sensors (twin-compared against the identical no-SOG/COG
streams, same rng draws): position RMSE improves 2.108 m -> 0.740 m
(constant-velocity) and 1.300 m -> 0.594 m (hydrodynamic); surge RMSE
improves 57% and 39% respectively. Sway is the interesting asymmetry:
76% improvement under constant velocity (SOG/COG are the *only*
observation of v there), but a small ~10% regression under the
hydrodynamic prior, where v is already pinned near the model's own
noise floor (~0.02 m/s) by the no-sway-forcing straight line, and
COG's measurement noise projected through dh/dv = u/s^2 costs a little
rather than helping. Position/heading do not regress under either
model, per the existing noisy-straight bounds.

RTK-class `GnssPositionCov` (2 cm std per axis, diagonal) against the
existing scalar 5 m path, same twin-streams technique: position RMSE
0.067 m vs 3.875 m (constant-velocity), 0.052 m vs 2.147 m
(hydrodynamic) -- comfortably past the order-of-magnitude-class bound
the regression test locks (5x floor, loose on purpose against the raw
~250x std ratio, since heading/gyro noise and the process model's own
budget floor the achievable improvement).

Low-speed segment (truth speed 0.02 m/s, well under the 0.5 m/s floor,
sandwiched between two 2 m/s cruise segments): COG is rejected
throughout the segment's steady-state tail under the hydrodynamic
prior, cleanly, because the prior's demand tracks the truth
deceleration at each instant. Under constant velocity the speed
*estimate* does not reliably settle back under the floor within the
scenario's window at all -- there is no deceleration dynamics to pull
it down, only the weak, indirect coupling through GNSS position fixes,
and SOG/COG (the direct correctors) stay locked out while the estimate
sits above the floor. This is a genuine dynamical property of the
constant-velocity model, not a test artifact: a self-reinforcing lockout
is possible in principle if the position-only channel is too weak to
pull the estimate down before the vessel accelerates again. Recorded
here rather than tuned away; the regression test only asserts the
universal safety properties for constant velocity (rejections happen,
no divergence, cruise-speed COG resumes once truth returns) and asserts
the full "converges then rejects cleanly" claim only where it actually
holds (hydrodynamic).

### Implementation notes

`coxswain-contract`: three new `MeasurementKind` variants
(`SpeedOverGround`, `CourseOverGround`, `GnssPositionCov`), a new
`GnssFixMode` enum, `EstimatorHealth::fix`. `coxswain-sim` gained
`VelocityModel` and `GnssCovModel` as new sensor kinds (`add_velocity`,
`add_gnss_cov`), following the existing `GnssModel`/`HeadingModel`
shape; `coxswain-keelson::publish_raw` maps SOG to
`speed_over_ground_knots`, COG to `course_over_ground_deg`, and
`GnssPositionCov` to `location_fix` with the real 2x2 covariance
(`PositionCovarianceType::Known`, distinct from the std-derived
`Approximated` the scalar paths use).
