## Phase 1: contract design

Representation choices for coxswain-contract, made against the manifest
schema v0.2 and the Fossen 3-DOF model:

- f64 for all physical quantities. Geodetic position does not survive f32,
  and the H753 has a double-precision FPU, so there is no case for mixing
  widths.
- Vessel state carries geodetic position (WGS84, radians) plus body-frame
  velocities (surge, sway, yaw rate). Covariance is 6x6 over
  [n, e, psi, u, v, r] in the local NED tangent frame: estimation happens
  in the tangent frame, reporting in geodetic.
- The contract crate stays dependency-free apart from optional serde.
  nalgebra lives in estimator/guidance; they convert at the boundary. The
  cost is a few from_column_slice calls, the gain is a contract that never
  moves because a math dependency did.
- Identities are integer newtypes (ClaimantId, SensorId, u16). Names map to
  ids at the adapter edge and in the manifest compiler; authority and
  fusion logic never parse strings.
- Actuator command is the generalized force demand (tau: surge, sway
  forces, yaw moment). Thrust allocation to physical actuators is post-MVP
  per D-021, so the contract does not guess its shape.
- Fixed-capacity BoundedList (hand-rolled, ~80 lines) instead of a heapless
  dependency: only config data needs it, and equality/serde semantics
  (compare and serialize the live prefix only) are easy to get subtly wrong
  in a generic dep wrapper anyway.
- Setpoint enum is deliberately minimal (Idle, HeadingSpeed, StationKeep);
  Phase 4 grows it with path following. Additions are review points.

Candidate for DECISIONS.md: the representation bundle (f64, frames,
dep-free contract, integer identities). Written as D-023.

CI green on GitHub for both the Phase 0 and Phase 1 pushes.

## Phase 2: estimator design

- Measurements carry their own noise std (contract MeasurementKind). The
  source declares what it knows: the simulator its configured noise, a GNSS
  driver its HDOP-derived guess. This defers schema open question 1
  (manifest noise parameters vs priority lists) per D-022 instead of
  answering it by accident. The heading priority list in EstimatorConfig is
  correspondingly unused for now: all licensed heading sensors fuse,
  weighted by reported std.
- EKF over [n, e, psi, u, v, r] in a local NED tangent frame anchored at
  the first accepted GNSS fix. Constant-velocity (constant-twist) process
  until the hydrodynamic prior lands in Phase 3. Euler discretization,
  analytic Jacobian, wrapped heading innovation.
- Process noise constants are documented placeholders until system
  identification (parked task). Replay bounds are deliberately loose.
- Licensing enforced at intake: a measurement fuses only if its sensor id
  is in the matching EstimatorConfig list and carries inner_loop license.
  Rejection is a typed result, not a silent drop.
- Health: Fault until initialized (first fix + first heading), Degraded on
  any stale fused role, else Nominal. Stds reported raw; covariance
  thresholds belong to the supervisor.
- Replay harness lives in coxswain-estimator/tests/ with a hand-rolled
  seeded xorshift RNG (determinism across platforms, no rand dep). JSONL is
  the recorded-log format until real recordings exist. Promote the harness
  to a crate only when another consumer appears.

## Phase 3 (started in parallel): coxswain-model

The model crate depends only on the contract, so it proceeds alongside the
estimator. Diagonal-M Fossen 3-DOF (the schema's param set has no coupling
coefficients, x_g = 0 assumed at the midship origin), SNAME sign
convention, linear damping. API serves both D-020 consumers: nu_dot +
analytic Jacobian for the estimator, RK4 step for the plant. Verification
is closed-form: terminal velocities, exponential time constants, Coriolis
skew-symmetry, passive energy decay, finite-difference Jacobian check, RK4
order check.

## Phase 2 and model results

Model verification numbers: terminal surge relative error 3.36e-4 at eight
time constants, which is exp(-8) itself, so the integrator adds essentially
nothing. RK4 error ratio 18.0 on dt halving (4th order predicts 16).

Estimator replay results (seeded, regression-locked in CI):

- Noise-free straight line: max position error 8 mm, heading 0.002 deg.
- Noisy straight line (GNSS 1 Hz / 2 m, heading 5 Hz / 1 deg, gyro 20 Hz /
  0.01 rad/s, 120 s): position RMSE 2.11 m against 2.83 m raw 2D fix
  noise, heading RMSE 0.29 deg, mean NEES 2.35.
- Turn through the +-pi wrap: heading RMSE 0.32 deg, no 2pi excursions.
- GNSS dropout: position std 4.0 m before the gap, 57 m after 23 s of
  silence, 3.3 m ten seconds after recovery; staleness flag and Degraded
  level flip exactly at max_age.
- An enrichment-licensed biased heading stream: 240/240 rejected, RMSE
  identical to the clean run.

Two honest bound adjustments, commented in the tests rather than tuned
away: position RMSE bound 2.5 m (the mandated 0.5 m/s^2 acceleration
budget runs conservative against a zero-acceleration truth) and NEES band
[1, 12] for the same reason. The filter is conservative, not
overconfident; the upper NEES bound still catches overconfidence
regressions. Process noise mapped as white-acceleration PSD (Q scales with
dt, not dt^2) so covariance growth per second is independent of tick rate.

LocalFrame (the geodetic-to-NED tangent frame) moved from the estimator
into coxswain-model: the simulator anchors the same local state to
geodetic truth, and two copies of geodesy is how the two consumers drift
apart. Same D-020 logic as the model itself.

Deferred contract addition: power/voltage telemetry. The simulator's
voltage sag fault needs a reading, but its consumer is the supervisor, so
the type is defined in Phase 4 with the failsafe matrix rather than
guessed now. ModelParams gains a ConstantVelocity variant with the
estimator promotion: sysid is parked, so the CV fallback stays selectable
per vessel.

## Phase 3 complete: hydrodynamic prior and simulator results

Every replay case now runs under both process models with unchanged
bounds. The promotion pays: noisy straight-line position RMSE 2.11 m
(constant velocity) vs 1.30 m (hydrodynamic), mean NEES 2.35 vs 1.98.
During the 23 s GNSS gap the predicted position std grows to 57 m under
constant velocity and 13.4 m under the prior with correct tau; maximum
in-gap position error 17.6 m vs 15.2 m. The prior knows the vessel
decelerates when the demand does.

The simulator emits contract measurements with configurable noise, bias,
latency (default zero; nonzero interacts with the estimator's
out-of-order rejection, revisit when drivers land), dropout,
quantization, and a settable bus voltage. Session-limit interruption
killed three agents mid-flight; the estimator and sim work survived
intact in the tree and verification passed after two clippy
large_enum_variant allows (inline waypoint list, precomputed model
matrices; boxing is not an option in no-alloc crates).

## Phase 4 design: failsafe matrix v1 and contract growth

Contract additions with their consumers now arriving: PowerStatus
(voltage into the failsafe matrix), Setpoint::FollowPath (16-waypoint
bound; paths are runtime claims, never manifest data), and a reserved
AUTONOMY claimant id so conn_grant_default has a target at boot.

Failsafe matrix v1, priority order, first match wins the setpoint:

1. Critical voltage: Idle + forced disarm. Driving on a dying battery
   ends worse than drifting.
2. Position degraded (estimator Fault, or GNSS stale past
   position_degraded_after): Idle, stay armed. Cannot hold station
   without a position; zero thrust beats driving blind.
3. Geofence breach: manifest action (hold at breach point, return to
   last inside position, or zero thrust), stay armed. Latched until back
   inside; the failsafe setpoint itself drives re-entry, so no extra
   hysteresis.
4. Claimant lost (heartbeat stale or release while armed): conn revoked,
   station-keep at the loss position, stay armed. This is the D-008
   story: the vessel holds station on its own authority.
5. Low voltage: report-only, holder keeps the conn and the setpoint;
   arming (but not staying armed) is refused. An operator limping home
   on a soft battery must not be overridden.

StationKeep targets latch at detection; a per-tick re-derived target
would drift with the vessel and defeat the point. Conn grant is
synchronous and non-preemptive in the MVP; GCS priority schemes are
post-MVP. Guidance will fold the control law (P + damping feedforward)
into itself for the MVP rather than splitting a controller service; the
supervisor's directive carries the effective setpoint either from the
holder or from the matrix.

## Phase 4: supervisor and guidance results

Supervisor: 37 tests including a table-driven pairwise priority matrix
over all failsafe cause combinations. Interpretations worth keeping:
pre-registered AUTONOMY is exempt from heartbeat staleness until its
first heartbeat (else the boot grant is pointless); staleness latches
ClaimantLost even while disarmed but a clean release latches only when
armed; Return with no inside position ever seen holds at the breach
point; failsafe causes evaluate and latch every tick regardless of
arming, the setpoint just stays Idle while disarmed.

Guidance closed-loop against the simulator (truth fed back, 100 ms
ticks): 90 deg heading step at 1 m/s settles to the 2 deg band in 18.6 s
with 3.7 percent overshoot; speed step settles in 5.4 s; station-keep
from 30 m is inside 3 m at 20 s and stays; dogleg max cross-track 4.3 m
against the 5 m bound. One real finding: crediting the plant's yaw
damping N_r in the derivative gain gave 21 percent overshoot, because at
cruise speed the sway-yaw Coriolis coupling (m_v - m_u) u v acts as
anti-damping of about the same magnitude. kd = 2 zeta wn (Izz - N_rdot)
with N_r left as margin fixed it. The far-zone station-keep approach law
clamp(0.3 d, 0, 1.5) saturates everywhere in the far zone, so the ramp
only matters through the near-zone handoff; noted, not changed.

## Phase 4 closed: full loop against the simulator

Core (in coxswain-hosted's lib for now; graduates to a core crate when
the H7 profile needs it) wires estimator -> supervisor -> guidance per
tick and feeds the demand back into the hydrodynamic prior. Six
scenarios run the honest loop, sim -> sensors -> estimator -> supervisor
-> guidance -> plant, no truth feedback anywhere:

- Claimant lost mid-transit: revoked one tick after the heartbeat bound,
  vessel stops and holds within 1.7 m of the detection point.
- GNSS gap: failsafe engages inside the configured window, demand zero,
  coasts to 0.03 m/s, resumes to 1.44 m/s after recovery.
- Critical voltage: disarm on the first tick that saw it, re-arm refused.
- Geofence Hold: latched 0.3 s before the truth crossing (estimate-led),
  overshoot 1.5 m past the 80 m fence line.
- Low voltage: report-only confirmed; arming refused, transit unaffected.
- D-008 rehearsal: all claimant traffic dropped during station-keep,
  vessel holds within 0.21 m over the final 30 s, conn revoked, core
  ticks well-formed with no claimants at all.

Finding worth a later decision: GeofenceAction::Hold latches the
station-keep target at the breach-detection point, which is by
construction just outside the ring, so a Hold breach never clears (the
re-entry comment in the supervisor holds for Return, not Hold). Hold
therefore means "stopped at the boundary, failsafe latched until an
operator acts", which is defensible; make it deliberate or retarget Hold
slightly inside in v2.

## Keelson conventions surveyed, adapter mapping decided

Verified against RISE-Maritime/keelson main (release 0.5.4): keys are
{base}/@v0/{entity}/pubsub/{subject}/{source_id} with @rpc for services,
payloads ride a protobuf Envelope (enclosed_at + bytes) with well-known
schemas per subject in subjects.yaml, and fused vs raw is distinguished
by source_id, not subject. Commands are deliberately RPCs, not subjects.
No Rust SDK exists, so the adapter vendors the handful of protos it needs
(Envelope, Primitives, foxglove LocationFix, EntityHealth) and compiles
them with prost.

Mapping for coxswain-keelson:

- Fused state: location_fix from the estimator (source_id
  coxswain/estimator), heading_true_north_deg as TimestampedFloat in
  degrees, plus speed subjects as they map cleanly.
- Raw licensed streams: same subjects, source_id coxswain/raw/{sensor}.
- Health: entity_health. Manifest hash + revision have no Keelson home,
  so they ride a coxswain-specific subject (unregistered upstream;
  candidate for a subjects.yaml PR later).
- Conn state: no Keelson claim/grant protocol exists (operational
  authority and roc_status are status reports, not arbitration). The
  claimant protocol becomes coxswain-specific RPCs under @rpc (register,
  request_conn, release_conn, arm, disarm) with small dedicated protos,
  and the holder's setpoint is a continuous pubsub stream that doubles as
  the heartbeat, matching Keelson's dead-man doctrine for control input
  (continuous publication, silence means stop). Documented as an
  interface to propose upstream once it has proven itself.

## Manifest compiler landed

TOML -> validate (all ten schema-doc rules as typed errors) -> postcard
blob with CRC32 and ed25519 signature -> no_std reader, 35 golden and
rejection tests including six tamper cases, Seahorse example verbatim
from the schema doc. Blob layout: magic, version, length, postcard
payload, CRC, signature; SHA-256 over the whole signed blob is the
published hash (D-024). Two D-018-driven scope widenings beyond the
first cut: conn_node (the watchdog interval governs behavior) and the
full authored sensor surface (lever arms, orientations, quirk tables)
ride inside the blob. Staleness defaults per role (gnss 3 s,
heading/compass 1 s, imu 0.5 s, other 5 s) with the 0183 quirk
max_age_ms as override: this is the estimator's Phase 2 answer landing
per D-022; a general per-sensor field is v0.3 business. One geometry
subtlety: a symmetric bowtie has exactly zero shoelace area, so the
degenerate-ring check fires before self-intersection; the tests use an
asymmetric bowtie for the intersection rule.

## README rewrite, and a boot finding from running the demo

Writing the user-facing README meant running the quick-start for real,
which surfaced two things. The manifest tool had no way to derive the
public key from a signing seed, so commissioning was impossible from the
CLI alone; a pubkey subcommand fixes it. And booting the unmodified
Seahorse example latched GeofenceBreach from the first tick: the hosted
profile anchored the sim at the first geofence vertex, a boundary point
where the inside test may resolve either way (here: outside). The origin
is now the ring's vertex centroid, safely inside the declared waters.
The integration tests had masked this by disabling the fence; the lesson
is the usual one, run the thing a user runs.

## RC hand controller brought into scope

Discussed an RC receiver as a physical takeover path, previously absent
from the plan except as an aside in the Phase 6 trial task. It slots in
rather than bolts on: the receiver terminates at the conn node on a UART
(D-009), a transmitter switch maps to the claimant verbs and the sticks
to setpoints (D-005), and link loss is the existing ClaimantLost path
since SBUS and CRSF carry explicit failsafe flags. The one real
architectural item is preemption, which Phase 4 deliberately parked:
takeover means taking the conn from autonomy without its cooperation.
Settled as per-claimant priority declared in the manifest, integers
compared in the supervisor, no special-cased RC path. Written as D-025.

Sequencing per the same entry: RC kill/disarm channel before RC as a
conn-holding claimant, both in Phase 6. Manual helm needs a direct-effort
Setpoint variant in the contract; RC bypasses guidance, never the
supervisor or arming. CRSF/ELRS is the preferred link, SBUS the fallback.

## Yaw-rate degradation experiment: does first water need an IMU driver?

Question from scoping the Phase 6 hardware list: can the inner-loop
sensor set come off 0183 instruments alone, or does the trial need a
dedicated IMU chip driver? Answered in the replay harness (temporary
test, deleted after; seeds 2 and 3 matching the existing straight-line
and turn cases; hydrodynamic model; GNSS 1 Hz / 2 m and heading
5 Hz / 1 deg held at baseline while the gyro stream varied).

- Gyro rate drives heading RMSE almost alone: 0.29 deg at 20 Hz, 0.37 at
  10 Hz, 1.55 at 1 Hz, 1.63 with no gyro at all (straight line; the turn
  reads 0.29 / 0.41 / 1.37 / 1.43). Position RMSE barely moves across
  every variant (1.24 to 1.52 m): GNSS carries it. Mean NEES stays in
  1.6 to 2.1 throughout, so the filter remains honest about its own
  uncertainty as the gyro degrades.
- Delivery latency is not a tuning question, it is a rejection question.
  The single-timeline out-of-order policy (estimator lib.rs:138) drops
  any measurement older than the last accepted one, so a 1 Hz gyro with
  200 ms latency loses 98-99% of its samples to the faster undelayed
  heading stream and is numerically identical to no gyro. Bus-sourced
  ROT at low rate is therefore worthless under the current policy; not
  worth routing around.
- The hydrodynamic prior carries roughly 40% of position accuracy once
  the gyro is gone: no-gyro position RMSE 2.11 m under constant velocity
  vs 1.24 m under the prior.
- A finding that matters regardless of hardware: no gyro plus heading
  degraded to 1 Hz diverges to NaN on the straight-line case. Genuine
  EKF instability, reproduced with an independent sequential feed: with
  1 s between corrections and no r observation, the Euler-discretized
  predict blows up (r estimate 5.5, 49, -3043, NaN inside 17 s). It is
  trajectory dependent (the turn case survives, badly, at 8.5 deg RMSE).
  The predict step needs substepping or a covariance safeguard before
  real sensors, and estimator health must flag NaN as Fault. Added to
  the Phase 6 backlog as a pre-water gate.

Conclusion for the hardware list: a single 0183 GNSS compass emitting
position and heading at about 5 Hz is enough for first water. Heading
RMSE around 1.5 deg sits inside guidance's 2 deg settle band at trial
speeds, and health/NEES stay sane. The IMU chip driver leaves the
critical path. Two conditions attach: heading at 5 Hz, not 1 Hz (the
divergence case), and the predict safeguard lands first.

## Five software-only Phase 6 tasks landed (subagent run)

Sequenced through sonnet subagents, each verified against the full gate
before the next started. 224 tests across the workspace, fmt/clippy/
thumbv7em green throughout.

- EKF predict safeguard: predict now substeps at 100 ms max (bit-exact
  identity for intervals at or under the nominal tick, verified to
  1e-12), and health reports Fault on any non-finite state or
  covariance element. The diary divergence case above is now a locked
  replay regression: finite throughout, heading RMSE 5.7 deg where it
  previously went NaN. Noted for later: measurement intake performs no
  finiteness validation, so a NaN from a real driver poisons the filter
  until the health gate catches it; rejecting non-finite values at
  intake is a small follow-up worth doing with the GNSS driver.
- D-025 implemented. SupervisorConfig carries manifest-declared
  (ClaimantId, u8) priorities, request_conn preempts on strictly
  greater, everything else refused as before. Preemption is a clean
  transfer: no ClaimantLost latch, arming untouched, ex-holder release
  is a no-op. Setpoint::DirectEffort reuses the ForceDemand type and
  passes through guidance's existing saturation. Manifest schema is
  v0.3 with an optional [[claimant]] table (name, id, priority),
  SCHEMA_VERSION bumped 1 to 2, duplicate ids rejected. One deliberate
  edge: DirectEffort has no Keelson wire representation and converting
  it returns a typed error, since RC is local to the conn node and a
  remote claimant streaming raw tau has no story yet.
- Driver trait in coxswain-drivers: init, self_test,
  read_with_timestamp(acquired_at) with associated Reading/Error types.
  Timestamping policy documented at crate level: stamps are acquisition
  time from a caller-injected monotonic clock, drivers never read a
  clock, a slow parse never moves the stamp. Blocking for now; the
  Embassy binding is a later phase.
- coxswain-nmea0183: zero-dep no_std strict parser, GGA/RMC/HDT/VTG,
  82-byte fixed buffer accumulator plus one-shot parse, checksum
  mandatory (the manifest's per-bus checksum mode is the one quirk).
  First cut rejected NMEA 2.3+ mode indicators, which every receiver
  since ~2002 emits; corrected to accept 11/12/13-field RMC and
  8/9-field VTG with the FAA mode surfaced as an enum, since an
  estimated or simulator fix must be distinguishable downstream.
  Seeded pseudo-fuzz (10k iterations) in normal cargo test.
- coxswain-crsf: zero-dep no_std sibling of the 0183 crate.
  RC_CHANNELS_PACKED (16x11-bit unpacking, code-point-to-microseconds
  helper) and LINK_STATISTICS (all ten fields, signed SNR). CRC8
  DVB-S2 checked against the published 0xBC test vector, golden frames
  built by an independent reimplementation in the tests. Unknown frame
  types with valid CRC are skipped rather than rejected because a live
  receiver interleaves telemetry; resync drops one byte, not the
  buffer, since CRSF has no sentence-start marker.
